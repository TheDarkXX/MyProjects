# Viral Planner Pro Enhancements

We have successfully implemented the four major requested components for Viral Planner Pro:
1. **Re-schedule/Repost Clips:** Users can click "🔁 Repost" on posted rows, which opens a native HTML modal to customize the caption and pick a schedule slot (or custom time) to clone and repost the clip.
2. **Interactive 3-Group Filter Fix:** Fixed the pipeline count bug to respect the selected page filter. Added an Active Filter Indicator strip with clearable chips, and enabled toggle-off states when clicking active filter states.
3. **Analytics Multi-Page Support:** Added page filter tabs on the Analytics view that query and filter database metrics, KPI totals, and charts dynamically per page.
4. **UX & Alert Fixes:** Replaced all native browser alert/confirm calls with non-blocking UI notifications (`showToast` and `vpConfirm` modal confirmations).

## Technical Details & Diffs

### 1. Repost Backend Route
Added `POST /planner/:id/repost` to duplicate a posted job under the same template but with a fresh status and custom caption/time.
```javascript
  app.post('/planner/:id/repost', async (c) => {
    try {
      const id = c.req.param('id');
      const original = db.prepare("SELECT * FROM mc_jobs WHERE id = ? AND job_type = 'viral'").get(id);
      if (!original) return c.json({ error: 'Job not found' }, 404);
      if (original.publish_status !== 'posted') return c.json({ error: 'Only posted jobs can be reposted' }, 400);

      const body = await c.req.json();
      const newCaption = body.viral_caption !== undefined ? body.viral_caption : original.viral_caption;
      const newScheduledAt = body.scheduled_at || null;
      const newPublishStatus = newScheduledAt ? 'queued' : 'unpublished';

      const result = db.prepare(`
        INSERT INTO mc_jobs (
          job_type, project_id, template_id, local_filename, story_id, viral_caption,
          page_name, local_file_path, viral_score, video_url, video_size, thumbnail_url,
          custom_thumbnail_url, active_thumbnail_type, title, status,
          approval_status, publish_status, scheduled_at
        ) VALUES (
          'viral', ?, ?, ?, ?, ?,
          ?, ?, ?, ?, ?, ?,
          ?, ?, ?, 'completed',
          'pending', ?, ?
        )
      `).run(
        original.project_id, original.template_id, original.local_filename, original.story_id, newCaption,
        original.page_name, original.local_file_path, original.viral_score, original.video_url, original.video_size, original.thumbnail_url,
        original.custom_thumbnail_url, original.active_thumbnail_type,
        `Repost: ${original.title || original.local_filename || 'Clip #' + id}`,
        newPublishStatus, newScheduledAt
      );

      const newJob = db.prepare('SELECT * FROM mc_jobs WHERE id = ?').get(result.lastInsertRowid);
      return c.json({ success: true, job: newJob }, 201);
    } catch (e) {
      console.error('[Repost Error]', e.message);
      return c.json({ error: e.message }, 500);
    }
  });
```

## Changelog

- **[2026-06-02] Added Repost Options (New Reel vs. Share Original):**
  Added selection options to the repost modal UI, allowing the user to select either:
  1. **Upload as New Reel (สร้างเป็น Reel ใหม่):** Clones the video job to post as a brand new Reel (triggers full resumable video upload).
  2. **Share Original Reel (แชร์จาก Reel เดิม):** Clones the job and records `is_share: true` in `input_variables`. When published, it executes a Graph API post to the page's `/feed` linking to the original Reel URL `https://www.facebook.com/watch/?v={original_fb_post_id}`. This aggregates views and earnings on the original Reel.
  The backend `/planner/:id/send-to-facebook` skips the file-based resumable video upload phase if `is_share` is true.

- **[2026-06-02] Fix "Session Expired" sync bug & Repost Modal "Invalid Date":**
  1. Updated Facebook Long-Lived Access Tokens directly in `fb_channels` via script on VPS to restore Graph API connectivity. Sync background jobs ran successfully and updated 60 stuck `scheduled` posts to `posted`.
  2. Fixed a bug in `brain-app-public/mission-control.js` where the Repost Modal slots logic expected an object (`s.utc`) but the `api('/mc/planner/next-slots')` returns an array of ISO date strings. This caused the new Repost modal to render "Invalid Date" for schedule slots. The fix handles both string and object structures to ensure forward/backward compatibility.
