# Codex MCP Integration for Antigravity

## 🎯 The Request
The user requested the ability for AG (Antigravity) to use the `oauth codex cli` to help review code (when Claude limits are reached) and to directly generate images from `gpt-image-2` within the AG interface. The user specifically wanted this to be a "Native" capability in AG.

## 🛠️ The Architecture (Native MCP)
Instead of using the VS Code Codex Plugin (which is strictly a UI tool for text generation), we built a custom **MCP Server** (`mcp_codex`) that exposes two tools directly to AG:
1. `gpt_review`: Submits code/text to `gpt-5.5` for review.
2. `gpt_imagine`: Generates images via `gpt-image-2` and saves them locally.

These tools do not use the `openclaw agent` CLI directly on the user's PC. Instead, they hit the Brain API (`https://brain.doctorbankonline.com/api/ai/chat` and `/api/ai/imagine`), which proxies the requests cleanly to the `ai-gateway.js` running on the VPS.

## 🚀 Backend Refactor (VPS)
We discovered that OpenClaw Gateway was running on port `18790` (not the default `18789`), and the `ai-gateway.js` (port `18810`) was already set up with an `/openai/` route.

**Fixes Applied:**
1. **Direct HTTP Routing:** Modified `routes/ai.js` in `brain-app`. Previously, `gpt-5.x` models were spawned via a heavy `child_process.exec` call to `openclaw agent`. We refactored this to use a standard `fetch` call to `127.0.0.1:18810/openai/v1/chat/completions`. This makes GPT responses significantly faster and more stable.
2. **Image Generation Route:** Added an intercept for `/openai/images/generations` in `ai-gateway.js` that translates standard OpenAI image requests into the specific Codex SSE `image_generation` tool format, extracts the `partial_image_b64`, and returns it cleanly.
3. **Brain API Endpoint:** Exposed `/api/ai/imagine` in `routes/ai.js` to allow the local MCP Server to securely request image generation.

## 📦 MCP Implementation
Created `C:\Users\Admin\.gemini\antigravity\mcp_codex\index.js` using the `@modelcontextprotocol/sdk`. 
- Modified `mcp_config.json` to point to this new node process.
- The tools interact over HTTPS to the Brain API, making the architecture robust and centralized.

1. VPS backend deployed via `git push vps`.
2. `ai-gateway` manually restarted via `pm2 restart ai-gateway`.
3. MCP Server installed and configured successfully.

## Changelog
- **Update 1:** Discovered that Codex API was rejecting `openai-codex/gpt-5.4`. We hotfixed `ai-gateway.js` to strip the `openai-codex/` prefix in `chatToResponses`.
- **Update 2:** Image generation was returning empty responses despite correctly parsing `partial_image_b64`. Discovered a catastrophic typo in `ai-gateway.js` line 125: `sseData.split('\\n')` which prevented JSON payload splitting. Hotfixed to `\n` and successfully generated the test infographic.

# GPT-5.4 Code Review: `ai-gateway.js`

จากการตรวจสอบโค้ด `ai-gateway.js` ด้วย **GPT-5.4** พบว่าตัว Gateway วางโครงสร้างการทำงานได้ดีครับ แต่อาจมีช่องโหว่ด้านประสิทธิภาพ (Performance) และความปลอดภัย (Security) หากมีปริมาณ Request เข้ามาเยอะพร้อมๆ กัน โดยมีจุดที่ควรปรับปรุง (High-priority issues) ดังนี้ครับ:

---

## 🚨 1. ปัญหาด้านหน่วยความจำ (Memory Leaks & Buffering)
- **Unbounded request/response buffering:** ระบบเก็บข้อมูล Request และ Response ทั้งก้อนไว้ใน Memory (`body.push(chunk)`) โดยไม่มีการจำกัดขนาด (`MAX_BODY_SIZE`) หากมีคนส่ง Payload ใหญ่ๆ (เช่นรูปภาพ) เข้ามา หรือ Response จาก AI ยาวมากๆ อาจทำให้ระบบค้าง หรือเกิด **Out of Memory (OOM)** ได้
- **Image SSE Buffering:** ใน `handleCodexImageRoute` คุณเก็บข้อมูล SSE ทั้งหมดเข้าตัวแปร String (`sseData += chunk.toString()`) ก่อนจะเริ่มแปลง ควรเปลี่ยนเป็น Incremental Parsing (อ่านและเคลียร์ไปทีละบรรทัด) เพื่อลดการใช้ Memory ครับ

## 🚨 2. ปัญหาการอ่านเขียนไฟล์ชนกัน (Race Conditions ใน `usage.json`)
- **Concurrency Issue:** ระบบอัปเดตไฟล์ `usage.json` ด้วย `fs.readFileSync` และตามด้วย `fs.writeFileSync` ในขณะที่มี Request พร้อมกัน 2-3 อันเข้ามา มันจะอ่านไฟล์เดิมไปทั้งคู่ และเซฟทับกัน ทำให้ยอด Token ที่ควรนับรวมกันเกิดอาการ **"ยอดหาย" (Lost updates)** ได้
- **Blocking File System:** การใช้ `*Sync` ทำให้ Node.js หยุดรอระหว่างที่กำลังเขียน/อ่านไฟล์ แนะนำให้ใช้ Async Queue, SQLite หรือ Append-only log แทน

## 🚨 3. การประมวลผล SSE ที่เปราะบาง (Fragile SSE Parsing)
- **SSE Chunk Split:** ข้อมูลที่มาเป็น Chunk บางครั้งบรรทัดอาจถูกตัดกลางคำ การใช้ `chunk.toString().split('\n')` โดยไม่เก็บ Buffer รอประโยคที่สมบูรณ์ อาจทำให้ JSON ขาดหาย และเกิด Error ได้
- **Silent Failures:** คุณใช้ `try { ... } catch (e) {}` ทำให้เวลา AI คืนข้อมูลแปลกๆ หรือรูปแบบเปลี่ยนไป โค้ดจะผ่านไปแบบเงียบๆ ทำให้ Debug ยากมาก

## 🚨 4. ปัญหาด้านเครือข่ายและความปลอดภัย (Network & Security)
- **Double Response Writes:** ในจังหวะที่ติด Timeout (`proxyReq.setTimeout`) คุณสั่ง `proxyReq.destroy()` ซึ่งมันจะไป Trigger Event `error` ซ้ำซ้อน ทำให้โค้ดพยายาม `res.writeHead` 2 รอบ (504 กับ 502) ซึ่งจะทำให้เซิร์ฟเวอร์แจ้งเตือน Error ทันที
- **Overly Permissive Headers:** ตัว Gateway Forward Headers เดิมไปทั้งหมด เช่นพวก `cookie`, `origin`, `x-forwarded-for` ซึ่งอาจทำให้ข้อมูลระบบของเราหลุดไปยัง 3rd Party ได้ ควรใช้วิธีระบุเฉพาะสิ่งที่จะส่งไป (Allowlist)
- **No Inbound Request Timeout:** หากผู้ใช้งานโหลดข้อมูลช้ามากๆ (Slowloris) ระบบจะรอรับข้อมูลไปเรื่อยๆ จนเต็ม

## 💡 จุดที่พบเพิ่มเติม
- ใน `chatToResponses` หากมี `system` message หลายอัน ตัวโค้ดปัจจุบันจะหยิบมาแต่อันสุดท้ายเท่านั้น (ควรนำมาต่อกัน)
- ปัญหา Model ของ OpenAI Image Gen ถูก Hardcode ว่าใช้ `gpt-5.4` ทำให้ส่งอากิวเมนต์อื่นๆ ไม่ได้
- `import path from 'path'` และ `import { execSync } from 'child_process'` ถูกประกาศไว้แต่ไม่มีการเรียกใช้ (Unused)
