# 🎯 Mission Control URL State Persistence

## 🎯 Goals & Context
When users navigate to the Mission Control page, they interact with different sub-views (Dashboard, Projects, Viral Planner Pro, Reel Blueprints). Previously, the parent SPA router (`app.js`) only tracked the main `#mc` hash, while the sub-view was stored in-memory (`mcView` inside `mission-control.js`).

As a result, refreshing the page or using direct links always reverted the user back to the default Mission Control Dashboard.

This update resolves this by mapping each sub-tab to its own unique URL hash (`#mc`, `#mc-projects`, `#viral-planner`, `#mc-reel-blueprints`), ensuring navigation states are preserved on reload.

---

## 🛠️ Changes Implemented

### 1. Main Navigation Router (`brain-app-public/app.js`)
- Added mappings for new sub-pages (`mc-projects`, `mc-reel-blueprints`) to the `PAGE_TO_CATEGORY` and `PAGE_TITLES` configuration.
- Extended the `navigateTo` function:
  - Mapped sub-tab targets so they correctly point to the `#page-mc` container element.
  - Dynamically triggers the appropriate rendering or initialization function (`initMissionControl()`, `mcShowProjects()`, `mcShowPlanner()`, `renderReelBlueprints()`).
  - Added background pre-fetching of Facebook channels if navigating to sub-pages directly without landing on the dashboard first.
- Updated `validPages` arrays inside both the browser `popstate` and initial `load` event listeners to whitelist the new sub-page hashes, preventing them from falling back to default pages.

### 2. Layout Structure (`brain-app-public/index.html`)
- Updated the sidebar menu item click triggers to call `navigateTo` directly with the target sub-page hash:
  - Dashboard: `navigateTo('mc')`
  - Projects: `navigateTo('mc-projects')`
  - Reel Blueprints: `navigateTo('mc-reel-blueprints')`

### 3. Sub-View Modules (`brain-app-public/mission-control.js`)
- Updated internal buttons (such as "← Dashboard", "View Projects →", "View All", "← Projects", and "← Back to List") to use `navigateTo` with the target sub-page hash instead of calling render functions directly.
- Ensures the URL hash updates seamlessly when navigating within the Mission Control area.

# 🎯 Reel Blueprint Pipeline Stabilization Complete

## 🚨 Root Cause of the Silent Failures

The issue "สำเร็จตรงไหนขึ้นเฟลหมดเลย" (UI shows success, but everything fails in the background with no data) was caused by a combination of two major factors in the background cron job:

1. **Free Tier Quotas & TPM Limits**: The `google/v1beta` endpoint for `gemini-2.5-pro` and `gemini-3.1-pro` was consistently hitting a hard `RESOURCE_EXHAUSTED` limit (quota 0).
2. **JSON Syntax Hallucinations**: When forced to use the smaller `gemini-2.5-flash` model to avoid quotas, the prompt complexity (a massive 6-panel array with 15+ fields per scene) overwhelmed the model's reasoning. It started outputting fundamentally broken JSON strings (e.g. `Unexpected character "o" at position 8538`) that could not even be salvaged by robust tools like `jsonrepair`. 

When these errors occurred, the background process caught the errors and updated the database status to `failed`, which explains why the UI queue succeeded but the actual rows ended up dead.

## ✅ The Fix: OpenRouter + Gemini 2.5 Pro

Instead of wrestling with the Google Free Tier API or writing complex regex fallbacks, I rerouted the `analyzeWithGPT` fetch request through the user's existing **OpenRouter** setup (`OPENROUTER_API_KEY`).

- **Model Used**: `google/gemini-2.5-pro`
- **Result**: By passing through OpenRouter, we utilize a premium connection that entirely bypasses the Google Free Tier hard-limits. `gemini-2.5-pro` is highly intelligent and flawlessly executed the massive JSON schema exactly as requested.
- **Data Sync**: The pipeline generated valid JSON, passed the `video_dna` check, synced successfully to Google Sheets, and cleanly updated the database status to `completed` with a `viral_score` of 38.

> [!NOTE]
> The OpenRouter integration is now live in `scripts/off-peak-agents/reel-blueprint-pipeline.js`. No local `.env` updates are required as it piggybacks on the key already configured for `discord-bot`.
