## Root Cause: Wrong Architecture

| Current | Problem |
|---------|---------|
| 3 agents × cheap models = 16 LLM calls/day | 16 bad calls > 1 good call |
| Inspector uses LLM to summarize SQL data | SQL results don't need LLM summarization |
| Ideator uses MiniMax M2.7 ($1.20/M) to architect | M2.7 can't read 14K char files, can't reason about architecture |
| 3 Personas (Bold/Pragmatic/Moonshot) with different models | All 3 produce identical output — persona ≠ personality at this model tier |
| Weekly Reflection uses Sonnet to distill rules | 1 rule in 23 days = the distillation model isn't the bottleneck, the input data is |
| Evolver writes code with M2.7 | Already killed — AG does this now |

**The math:**
```
Current:  16 calls/day × $0.0025/call × 30 days = $1.20/month → 3 useful proposals (3.3%)
Proposed: 1 call/day  × $0.15/call  × 30 days = $4.50/month → target: 15+ useful proposals (50%+)
```

Cost increases 3.75× but if yield goes from 3.3% to 50% that's a **15× improvement in useful output**.

## New Architecture: 2 Components Only

### Component 1: Deterministic Collector (Zero LLM)

**What it replaces:** Inspector (36,877 bytes) + all sub-inspectors + github-scanner

**What it does:** Pure SQL + filesystem checks. No AI. No hallucination possible.

```
Runs: 1×/day at 04:00 (single cron)
Runtime: <10 seconds (SQL only)
Cost: $0.00
Output: JSON report file at /root/brain-app/data/daily-intelligence.json
```

**Data it collects (deterministic, zero LLM):**

```sql
-- 1. System Health
SELECT name, pm2_env.restart_time FROM pm2_jlist;
SELECT COUNT(*) FROM watchdog_logs WHERE created_at > datetime('now', '-24 hours') AND result = 'FAIL';
SELECT cron_id, last_status, last_run FROM cc_crons WHERE enabled = 1 AND last_status = 'error';

-- 2. Business
SELECT COUNT(*) FROM cc_bills WHERE status = 'unpaid' AND due_date < date('now', '+7 days');
SELECT category, COUNT(*) FROM cc_expenses WHERE category IS NULL OR category = '' GROUP BY category;
SELECT COUNT(*) FROM cc_subscriptions WHERE end_date BETWEEN date('now') AND date('now', '+30 days');

-- 3. Content & Productivity  
SELECT COUNT(*) FROM content_plans WHERE status = 'idea' AND updated_at < datetime('now', '-14 days');
SELECT COUNT(*) FROM tasks WHERE status = 'todo' AND created_at < datetime('now', '-7 days');

-- 4. Hydra Pipeline Health
SELECT status, COUNT(*) FROM evolution_log WHERE created_at > datetime('now', '-7 days') GROUP BY status;
SELECT COUNT(*) FROM intelligence_pool WHERE status = 'open';

-- 5. Disk/DB Health
SELECT page_count * page_size as size_bytes FROM pragma_page_count(), pragma_page_size();
-- Check WAL size, backup integrity
```

**Output format:**

```json
{
  "date": "2026-05-06",
  "system": {
    "pm2_restarts": {"brain-app": 1101, "discord-bot": 1034, "ai-gateway": 41},
    "watchdog_fails_24h": 3,
    "cron_errors": ["cron-news", "cron-auto-planner"],
    "db_size_mb": 42.5,
    "backup_last": "2026-05-06T04:00:01Z",
    "backup_integrity": "ok"
  },
  "business": {
    "bills_due_7d": 2,
    "expenses_uncategorized_pct": 40,
    "subs_expiring_30d": 5
  },
  "content": {
    "ideas_stuck_14d": 3,
    "tasks_stale_7d": 12
  },
  "pipeline": {
    "proposals_7d": {"deployed": 2, "rejected": 5, "pending": 3},
    "pool_open": 18,
    "approval_rate_30d": "11%"
  }
}
```

**Why zero LLM:** Inspector's current LLM call is used to "summarize findings in Thai." That summary is never read by anyone — the data goes into a DB table. Raw numbers are more useful than AI summaries of numbers.

### Component 2: Daily Synthesis (1 × Opus-class call)

**What it replaces:** Ideator (3 personas), Weekly Reflection, GitHub Scanner

**What it does:** One high-intelligence call per day. Reads the full `daily-intelligence.json` + relevant source code + conversation history. Produces 0-2 proposals (quality over quantity).

```
Runs: 1×/day at 05:00 (after Collector)
Model: Claude Opus 4.6 or equivalent (same tier AG uses)
Context window: Full — no 8000 char limit
Cost: ~$0.10-0.20/call ($3-6/month)
Output: 0-2 proposals as Markdown stubs in _incoming/hydra/
```

**Key differences from current Ideator:**

| Current Ideator | New Synthesis |
|----------------|---------------|
| MiniMax M2.7 ($1.20/M) | Opus 4.6 ($15/M) |
| 8,000 char context limit | Full file reads (no limit) |
| 7 calls/day (3 personas × schedules) | 1 call/day |
| No conversation history | Reads last 3 days of AG conversation logs |
| 91 proposals in 23 days = 4/day | Target: 0-2/day (only if genuinely useful) |
| Output: always produces a proposal | Output: `null` is valid — "nothing to propose today" |

**Prompt structure:**

```
You are the Daily Synthesis agent for a production Node.js system.

RULES:
1. Output NULL if nothing warrants a proposal. Silence > noise.
2. Never propose something that already exists in the codebase. 
   You have full source code access — READ IT before proposing.
3. Never propose the same thing twice. You have the full evolution_log.
4. Think at the "Steve Jobs" level: does this change the ceiling of what's possible?
   If not, don't propose it.
5. Cross-system insights only. Single-file patches are not worth your time.

INPUT:
- daily-intelligence.json (today's system state)
- evolution_log last 30 entries (what's been proposed/rejected/deployed)
- Full source of key files (ai-call.js, server.js, routes/*, etc.)
- Last 3 days of AG conversation summaries
- Current Quick Save Active/ plans

OUTPUT (if any):
{
  "proposal_id": "SYN-YYYYMMDD-NNN",
  "title": "...",
  "why_now": "What data point triggered this?",
  "what_changes": "Concrete file changes",
  "what_it_unlocks": "What becomes possible after this that wasn't before?",
  "estimated_ag_effort": "1 session / 2 sessions / multi-day"
}
```

**The "null is valid" rule is critical.** Current Ideator always produces output because the prompt says "generate a proposal." This creates pressure to fabricate. The new system explicitly rewards silence.
