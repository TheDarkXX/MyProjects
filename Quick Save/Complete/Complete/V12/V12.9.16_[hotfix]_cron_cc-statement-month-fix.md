# V12.9.16 Fix CC Statement Old Month Bug

## Root Cause
When the CC statement cron script runs and extracts data using the AI, it determines the `statement_month` using the `buildExtractionPrompt`. For billing cycles that span across two months (e.g. 11 May to 10 Jun), the AI previously extracted the older month (e.g. May) because the majority of the transactions occurred in that month, resulting in the Discord bot reporting the statement as an old month. Furthermore, if the AI failed to provide a valid `statement_month`, the `defaultYearMonth` fallback logic sorted by transaction frequency, inherently selecting the older month with the most days in the cycle.

## Changes Made
1. **AI Prompt Refinement (`discord-bot/lib/credit-card-agent.js`)**: Updated the `statement_month` extraction prompt to explicitly instruct the AI to use the month of the statement's END date or ISSUE date.
2. **Fallback Calculation Fix (`discord-bot/lib/credit-card-agent.js`)**: Changed the `defaultYearMonth` fallback calculation from using the most frequent transaction month to using the newest transaction month to correctly capture the end of the billing cycle.

## Changelog
- **Replaced AI Extraction with Deterministic Logic**: As suggested by the user, the AI's output for `statement_month` is now completely overridden by a highly deterministic logic based on the Gmail email `Date` header.
- **Screener Logic**: If the email date is before the 15th of the month, the `statement_month` is strictly set to the previous month. If it is the 15th or later, it is set to the current month.
- **Why this is better**: This eliminates all AI hallucinations regarding the statement cycle. Even if the user manually re-runs the script on an old email (e.g. from May 13), the logic accurately extracts the actual email delivery date and correctly calculates it as `2026-04` instead of confusing it with the current execution time.
