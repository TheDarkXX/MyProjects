# Superpowers Iron Rules Cherry-Pick

## 📌 Context (Compiled Truth)
We reviewed the `obra/superpowers` repository (198k stars), an agentic skills framework. 

**Decision:**
- **Overall Verdict:** ICEBOX. The framework relies on greenfield architecture (Git worktrees, mandatory TDD without exception, 1-question-per-message brainstorming) which conflicts directly with our production "hotfix and deploy" VPS environment.
- **Action Taken:** We cherry-picked the 5 most valuable concepts that do not conflict with our workflow and integrated them into `AGENTS.md` to harden the system's behavioral integrity.

**The 5 Cherry-Picked Concepts Integrated into AGENTS.md:**
1. **No Placeholders Rule (Implementation Plans):** Agents must never write "TBD", "TODO", or "implement later". Every plan must contain the actual required content, exact file paths, and complete code snippets.
2. **Verification Gate Function (IDENTIFY→RUN→READ→VERIFY→CLAIM):** An upgrade to Iron Rule #2. Agents must empirically prove a claim by running the verification command first.
3. **Rationalization Table:** Added to the Verification Rule to counter common AI rationalizations (e.g., "should work now", "partial check looks good") which are now explicitly marked as `Not Done`.
4. **3-Strike Architecture Rule:** Added to the Dissent Rule. If an agent tries to fix a bug 3 times and fails, it must STOP attempting fixes and instead question the foundational architecture.
5. **Red Flags Pattern:** A self-check checklist for agents to monitor their own thoughts (e.g., "quick fix for now", "just try changing X") to prevent rule violations.

## ✅ ข้อดี (Pros)

### 1. 🧠 Verification Before Completion — *เราควร adopt*
เกือบ 1:1 กับ Iron Rule #2 ของเราที่ว่า "VERIFY BEFORE DONE" แต่ Superpowers ทำได้ละเอียดกว่า:
- มี **Gate Function** ที่ชัดเจน: IDENTIFY → RUN → READ → VERIFY → CLAIM
- มี **Rationalization Table** ป้องกัน agent ใช้คำว่า "should work", "probably", "seems to"
- มี **Red Flags list** ที่ trigger self-check

> **Adoptable?** ✅ เอา Gate Function + Rationalization Table มาเสริม Iron Rule #2 ได้เลย

### 2. 🔍 Systematic Debugging 4-Phase — *น่า adopt บางส่วน*
ตรงกับ philosophy ของเราแต่ทำได้เป็นระบบกว่า:
- **Phase 1:** Root Cause Investigation (read errors → reproduce → check changes → trace data flow)
- **Phase 2:** Pattern Analysis (find working examples → compare)
- **Phase 3:** Hypothesis & Testing (scientific method, ONE variable at a time)
- **Phase 4:** Implementation (failing test → single fix → verify)
- **Phase 4.5:** ถ้า fix 3+ ครั้งแล้วยังไม่ได้ → **Question the Architecture** ← เจ๋งมาก!

> **Adoptable?** ✅ Phase 4.5 "3-strike architecture question" ดีมาก ควร adopt

### 3. 🤖 Subagent-Driven Development — *น่าสนใจ แต่ยังใช้ไม่ได้*
Fresh subagent per task + 2-stage review (spec compliance → code quality):
- ป้องกัน context pollution
- Model selection ตาม task complexity (cheap model for mechanical, capable for architecture)
- Implementer status handling (DONE, DONE_WITH_CONCERNS, NEEDS_CONTEXT, BLOCKED)

> **Adoptable?** ⏸️ ยังใช้ไม่ได้ในระบบเราเพราะเราไม่มี subagent dispatch infrastructure แบบ Claude Code. ดีใน concept.

### 4. 📝 Writing Plans — *bite-sized task granularity*
แต่ละ step = 2-5 นาที, มี exact file paths, complete code, verification commands:
- **No Placeholders Rule:** ห้าม TBD, TODO, "implement later", "similar to Task N"
- **Self-Review Checklist:** spec coverage, placeholder scan, type consistency

> **Adoptable?** ✅ "No Placeholders" rule ดีมาก — เสริม implementation plan ของเราได้

### 5. 🛡️ Skill TDD (Meta) — *writing-skills*
ใช้ TDD สำหรับเขียน **documentation/skills เอง**:
- RED: Run scenario without skill → document what agent does wrong
- GREEN: Write skill → verify agent now complies
- REFACTOR: Close rationalization loopholes

> **Adoptable?** ⏸️ น่าสนใจแต่ scope ใหญ่ ต้องเปลี่ยนวิธี create skills ทั้งหมด

## ❌ ข้อเสีย (Cons)

### 1. 🚫 Brainstorming Gate ขัดกับ workflow เรา
- บังคับ **1 question per message** ก่อนเขียน code → ทำให้ hotfix ที่ควรจะเสร็จใน 5 นาทีกลายเป็น 30 นาที
- บังคับเขียน **design doc** ก่อนทุก task ("Even a todo list" quote) → overkill สำหรับ R2 tasks
- เรามี R0/R1/R2 classification อยู่แล้วที่แยก scope ได้ดีกว่า

### 2. 🚫 TDD Iron Law ขัดกับ production reality
- "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST" + "Write code before test? **Delete it.**"
- ระบบเรา 80%+ ไม่มี test framework (Discord bot, VPS cron, Express routes, shell scripts)
- Superpowers ออกแบบสำหรับ TypeScript/Python projects ที่มี Jest/pytest
- บังคับ TDD ทุกกรณีจะ break workflow ของเราทั้งหมด

### 3. 🚫 Git Worktree assumption
- ทุก task ต้องสร้าง worktree, branch, PR → ไม่เหมาะกับ `git push vps` deploy model ของเรา
- เราใช้ single branch + direct push → Superpowers workflow ไม่ match

### 4. 🚫 Too much ceremony สำหรับ hotfix workflow
- Pipeline: brainstorm → spec → review spec → plan → review plan → implement → spec review → code review → finish branch
- **Minimum 7 steps** ก่อน code ถูก deploy
- เราทำ hotfix: identify → fix → verify → push vps → done (4 steps)

### 5. 🚫 ไม่มี concept ของ "running production system"
- ไม่มี deployment flow, cache busting, VPS SSH, service restart
- ออกแบบสำหรับ "build from scratch" projects exclusively

### 6. 🚫 Claude-centric architecture
- ใช้ Claude-specific features (TodoWrite, subagent dispatch, `/plugin install`)
- Port ไป Gemini/AG ต้อง rewrite significant portions
- Prompt engineering tuned สำหรับ Claude's rationalization patterns

## 🎯 Verdict & Recommendation

1. Framework นี้ออกแบบสำหรับ **greenfield development** (เริ่มจากศูนย์) — ระบบเราเป็น **running production** ที่ต้อง hotfix ทันที

3. **Architecture ไม่ compatible** (git worktree, TDD-only, Claude-specific)
4. **Ceremony มากเกินไป** สำหรับ hotfix-heavy workflow ของเรา

- ไม่ adopt wholesale — architecture ต่างกันเกินไป
- Cherry-pick 5 concepts ที่ดีที่สุดมาเสริม AGENTS.md ได้เมื่อพร้อม
- กลับมาดูอีกครั้งเมื่อระบบเรามี test framework ครอบคลุมมากขึ้น

### ✅ เรื่องที่ Superpowers ทำดีกว่าเรา (ควรเรียนรู้):
