# V12.19.4 — Wire Redis Cache Reconnect on Read/Write

## 📌 Context (Compiled Truth)

### Root Cause
`lib/ai-call.js` exports a singleton `RedisCache`. Its error handler sets `this.connected = false` on socket close. However, `get()` and `set()` immediately `return` when `!this.connected || !this.client` instead of attempting reconnection through the already-existing `checkConnection()` / `_connectWithRetry()` methods.

### Impact
- Once Redis closes the socket (e.g., during deploy-hook restarts), AI response caching degrades to **permanent no-cache** for that process lifetime
- Increases duplicate AI gateway API calls and latency
- Only emits noisy logs, no self-healing

### Implementation Plan

1. **Modify `RedisCache.get()`:** Instead of early-returning on `!this.connected`, attempt `await this.checkConnection()` with a cooldown (e.g., max 1 retry per 30s) before giving up
2. **Modify `RedisCache.set()`:** Same pattern — attempt reconnect before early-return
3. **Add reconnect cooldown:** Track `lastReconnectAttempt` timestamp. If < 30s since last attempt, skip reconnect and return null/no-op (prevents reconnect storms during sustained outages)
4. **Suppress noisy empty Redis error logs:** Replace blank `[RedisCache] Redis client error:` with a stable coded message like `[RedisCache] Socket closed, will retry on next cache operation`
5. **Test:** Kill Redis briefly, verify cache reconnects on next AI call
