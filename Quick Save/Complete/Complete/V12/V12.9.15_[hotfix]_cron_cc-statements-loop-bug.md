# V12.9.15 Cron CC Statement Loop Bug Fix

## Description
The user reported that the `cc` script runs every day. 

Investigation showed that the cron is designed to run from the 15th to the 30th of every month. It attempts to stop running and "skip" execution if it detects that 4 unique banks have been imported for the current month.

The buggy logic:
```javascript
if (item.statement_month === currentYearMonth && item.credit_card)
```

Because KBank and SCB statements arrive in mid-May but actually cover April's billing cycle, the AI properly tags their `statement_month` as `2026-04`. As a result, in `2026-05`, the script only ever counts 2 banks (BBL and Krungsri) that have `statement_month === '2026-05'`. It never reaches the threshold of 4, and proceeds to run every day until the end of the month.

## Fix
Changed the completion-check logic to use the `created_at` timestamp rather than `statement_month`:
```javascript
if (item.created_at && item.created_at.startsWith(currentYearMonth) && item.credit_card)
```

This ensures the script checks if 4 unique bank statements have been *successfully imported during the current calendar month*, regardless of what billing cycle (`statement_month`) the bank was assigned by the AI.

## 🛠️ Master Reference: PDF Extraction Pipelines
Each bank is extracted differently due to variations in their statement formats.

1. **SCB / CardX (`pdftotext -layout`)**: Uses Layout Text parser. Standard text parsing clumps prices at the bottom of the page; `-layout` preserves the visual spacing so AI can read the columns accurately.
2. **KBank (`pdf-parse`)**: Uses Standard Text Parser. The format is clean enough to be read sequentially.
3. **BBL & Krungsri (`pdftoppm -jpeg`)**: Uses OCR / Vision Model. The tables are extremely complex to parse as text, so the PDF is converted to 150dpi JPEG images. The images are then passed to `qwen2.5-vl-72b-instruct` (Vision Mode) to be read like a human.
