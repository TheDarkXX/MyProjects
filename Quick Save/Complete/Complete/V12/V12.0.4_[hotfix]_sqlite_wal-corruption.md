### Gap 1: `db/brain.db` ยังถูก Git Track อยู่บน VPS ⛔

> [!CAUTION]
> นี่คือ **ระเบิดเวลา** ที่สำคัญที่สุด — ถึงแม้ไฟล์จะอยู่ใน `.gitignore` แล้ว แต่มันเคยถูก `git add` ไปก่อนหน้า ทำให้ `.gitignore` **ไม่มีผล** กับมันเลย

### Phase 3: Deploy Hook — เพิ่ม Row Count Guard

#### [MODIFY] [tmp/post-receive](file:///c:/My%20Claw/Openclaw-VPS/tmp/post-receive)

เพิ่ม **post-deploy row count comparison** เข้าไปใน hook:
```bash
# After integrity check passes, compare row counts
PRE_TASKS=$(cat /tmp/brain-db-pre-deploy-tasks 2>/dev/null || echo "0")
POST_TASKS=$(sqlite3 "$WORK_TREE/db/brain.db" "SELECT COUNT(*) FROM tasks;" 2>/dev/null)
DROP=$((PRE_TASKS - POST_TASKS))
if [ "$DROP" -gt 5 ]; then
  echo "[DEPLOY HOOK] 🚨 TASK COUNT DROPPED BY $DROP DURING DEPLOY!"
  echo "[DEPLOY HOOK] 🚨 Restoring from pre-deploy state..."
  # Auto-restore
fi
```

Flow ใหม่ของ Deploy Hook:
1. `pm2 stop brain-app`
2. `PRAGMA wal_checkpoint(TRUNCATE)` — flush WAL
3. **บันทึก row counts ก่อน deploy** (tasks, bills, expenses)
4. `git checkout -f`
5. **เปรียบเทียบ row counts หลัง deploy** → ถ้าลดลง > 5 → auto-restore
6. `PRAGMA integrity_check` → ถ้า fail → auto-restore
7. `pm2 start brain-app`

## 🐛 Root Cause Analysis

The user reported that "tasks that were done came back" and "bills and subscriptions reverted".
Investigation showed that the SQLite database (`brain.db`) was literally missing recent transactions (e.g., tasks completed in the last week, bills marked as paid).

The root cause was isolated to the Git `post-receive` hook on the production VPS:
1. The `post-receive` hook contained logic to copy `db/brain.db` and its WAL files (`brain.db-wal`) as a pre-deploy backup.
2. It then executed `git checkout -f`.
3. Finally, it restored the WAL file using `cp` **while the PM2 Node.js process (and thus SQLite) was still actively running**.
4. Truncating and replacing an actively used WAL file violates SQLite's integrity assumptions. When the PM2 process subsequently restarted, SQLite detected WAL corruption and was forced to perform a graceful rollback, discarding all transactions in the WAL that had not yet been checkpointed to `brain.db`.
5. Since the application had not been restarted in over 10 days, the WAL file contained up to 10 days of user activity. All of this activity was lost during the deploy.

Additionally, the recent EVO integration of `xxhash` for request coalescing was deployed without `package.json` updates, causing a `MODULE_NOT_FOUND` crash loop on the server immediately after deploy.

## ✅ Resolution

1. **Removed WAL Backup/Restore Logic**: Edited `/root/repos/brain-app.git/hooks/post-receive` on the VPS to remove the destructive SQLite backup and restore blocks. Since `db/brain.db` is correctly listed in `.gitignore` and untracked, `git checkout -f` will safely ignore the database entirely, rendering the manual backup unnecessary and dangerous.
2. **Fixed Server Crash**: Added `xxhash`, `xxhash-wasm`, and `redis` to `package.json` and executed `npm install` directly on the VPS to resolve the 502 Bad Gateway crash loop.
3. **Data Restored**: Smart-merged from daily backup (`brain_20260504_040001.db`) back into the live database. Restored 125 done tasks (from 21), 17 bills (from 9), 5 routines (from 0), 136 intelligence_pool entries (from 30), and 112 expenses (from 100).

## 🛡️ Database Fortress — 6 Layers (100% Coverage)

During deep inspection, it was found that the deploy hook alone wasn't enough because `db/brain.db` was actually still tracked by Git on the VPS index, acting as a ticking time bomb. The protection was upgraded to a 6-layer fortress:

### Layer 1: Git Untrack (The Core Fix)
- Executed `git rm --cached db/brain.db db/brain.db-wal db/brain.db-shm` and `db/backups/` directly on the VPS bare repo.
- Fixed BOM corruption in `.gitignore` to ensure it applies correctly.
- `git checkout -f` during deploy will now genuinely ignore the database files forever.

### Layer 2: Hardened Deploy Hook V2 (`post-receive`)
- **Pre-deploy**: `pm2 stop brain-app` → `PRAGMA wal_checkpoint(TRUNCATE)` flushes all WAL data to main DB before any file operations. Records task and bill row counts to temp files.
- **Post-deploy**: Calculates the difference in row counts. If task count drops by >5 or bill count by >2, or if `PRAGMA integrity_check` fails, it **automatically restores from the latest daily backup**.
- **Sanitization Bugfix**: Added `PRE_TASKS=${PRE_TASKS:-0}` to explicitly handle empty strings from sqlite counts, which previously caused bash arithmetic `$(())` fatal errors.

### Layer 3: Active Watchdog (`data-integrity-watchdog.cjs`)
- Upgraded the 6-hour watchdog script. Instead of just checking file size, it now connects via `better-sqlite3`.
- Checks `PRAGMA integrity_check`.
- Compares row counts against a sticky baseline stored in `.row-baseline.json`. If rows drop beyond thresholds, alerts Discord immediately. Updates baseline automatically if data grows.
- **Bugfixes**: 
  - Stopped `readFileSync` as UTF-8 on binary `brain.db` which wasted memory and could crash.
  - Handled `JSON.parse` failures on the baseline file so corrupt baselines don't silently reset the watchdog without alerting.

### Layer 4: WAL Checkpoint Cron
- Created a server crontab: `0 0,4,8,12,16,20 * * * sqlite3 /root/brain-app/db/brain.db "PRAGMA wal_checkpoint(PASSIVE);"`
- Runs every 4 hours to prevent massive WAL files (10 days of buildup) in the future.

### Layer 5: Enhanced Daily Backup (`daily-backup.sh`)
- Retention extended from 7 → **30 days**.
- Uses `sqlite3 .backup` (safe online backup API, not raw `cp`).

### Layer 6: Git Snapshot Time-Machine (`snapshot-backup.sh`)
- Daily git-based snapshots at `/root/openclaw-snapshots/`.
