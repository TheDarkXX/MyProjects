# 🧠 Oracle / AI Gateway CLI to MCP Refactor

## 🎯 Problem Statement
The user reported that the `Oracle` agent had been completely silent for 2-3 days, offering zero optimization or innovation proposals.

**Investigation Findings:**
1. **The Crash:** Oracle was not quiet due to stability; it was crashing immediately upon execution with: 
   `GPT-5.5 CLI failed: GatewayClientRequestError: FailoverError: Your authentication token has been invalidated.`
2. **The Root Cause:** In `V12.7.3`, the user implemented an MCP Server (`mcp_codex`) to route GPT requests directly to a local `ai-gateway.js` proxy (port 18810) using a new "ChatGPT Business Plan" OAuth token (`/root/.codex/auth.json`).
3. **The Missed Refactor:** While the Brain Web App (`routes/ai.js`) was updated to use this new HTTP route, **Oracle** (`scripts/hydra/oracle.js`) and the **Discord Bot** (`discord-bot/lib/ai.js`) were accidentally left behind. They were still spawning the `openclaw agent` CLI using the old, invalidated `GATEWAY_AUTH_TOKEN` from `.env`.

## 🛠️ The Solution: Direct HTTP Routing

To fix this and unify the AI pipeline, we completely removed the `child_process.exec` dependency on the OpenClaw CLI for backend agents:

### 1. `scripts/hydra/oracle.js`
Refactored `callGPT55()` to use a native `fetch()` call to `http://127.0.0.1:18810/openai/v1/chat/completions`, attaching the `X-Agent-Id: oracle-ai` header.

### 2. `discord-bot/lib/ai.js`
Refactored `callOpenClaw()` to also use the same `fetch()` logic. Added logic to auto-prefix `openai-codex/` to models if omitted, ensuring the local `ai-gateway.js` routes them correctly to the Codex API.

## ✅ Verification
1. Code committed and `node bump-cache.js` executed.
2. Pushed to VPS (`git push vps`).
3. Manually triggered Oracle on the VPS. It successfully connected to the GPT-5.5 model via the new MCP gateway, used tools (gitLog, readOpsEvents), and completed its loop without token errors.
