# Oracle Memory System — Complete Leak Audit

## Root Cause Analysis: 7 รูรั่วที่ทำให้ Oracle "ขี้ลืม / ตาบอด / สมองน้อย"

### 🔴 รูที่ 1: `tools_used: 0` — สมองน้อย ไม่ยอมตรวจสอบ
**Oracle ไม่เคยใช้ tool เลยแม้แต่ตัวเดียว!** มันสร้าง 6 findings จาก PM2 status string ตัวเดียวที่ให้ไปใน initial context โดยไม่เคยอ่าน log, ไม่เคย query DB, ไม่เคยดู git diff เลย — แปลว่ามันแค่ **เดาจาก context เล็กๆ** แล้วเขียน findings ออกมาทันที

**Fix:** เพิ่ม instruction ใน System Prompt: `MANDATORY INVESTIGATION: You MUST use at least 2 tools before producing findings.`

### 🔴 รูที่ 2: ไม่มี History ใน Prompt — ขี้ลืม
**Fix:** Inject `## Your Previous Proposals` จาก `evolution_log` ลงใน initial context

### 🔴 รูที่ 3: ไม่มี Feedback Loop — ไม่รู้ว่าอะไร reject/icebox
**Fix:** Inject `## Feedback from Human` แยก approved/rejected/iceboxed

### 🟠 รูที่ 4: `readQuickSave` ตาบอด — อ่านแค่ `Complete/`
**Fix:** เปลี่ยน `readQuickSave` tool ให้อ่านจาก `Complete/`, `Active/`, และ `_incoming/hydra/`

### 🟠 รูที่ 5: `full` vs `delta` mode ไม่ต่างกันจริง
**Fix:** `delta` mode ถูกสั่งผ่าน prompt ให้ focus ONLY on changes since the last run.

### 🟡 รูที่ 6: ไม่เห็น `ops_events` ledger
**Fix:** เพิ่ม tool `readOpsEvents` และ inject ops_events ล่าสุดลงใน initial context

### 🟡 รูที่ 7: Dedup ทำงานที่ output-level เท่านั้น
**Fix:** แก้ที่ **input-level** (ใส่ history ใน prompt) ไม่ใช่แค่ output-level

## Implementation Details

All changes were implemented in `scripts/hydra/oracle.js`:
- Added `buildMemoryContext` to retrieve data from SQLite (`evolution_log`, `ops_events`, `oracle_reports`).
- Expanded `readQuickSave` to pull from 3 core directories.
- Added `readOpsEvents` as a dedicated tool.
- Hardened `SYSTEM_PROMPT` with `CRITICAL MEMORY RULES` and `MANDATORY INVESTIGATION`.
- Updated `processFindings` to store `recent_topics` and `rejection_memory` inside `oracle-profile.json`.

**Verification:** Syntax checked, deployed to VPS. Will monitor next Oracle run to ensure `tools_used > 0` and duplicate findings cease.
