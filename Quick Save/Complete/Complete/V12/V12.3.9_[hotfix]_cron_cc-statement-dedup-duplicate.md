# 🚀 Implementation: Fix CC Statement Deduplication Ignoring Identical Transactions

## 🎯 Problem Statement
The user reported that the `cc-statement` agent was missing identical transactions that occurred consecutively (e.g., buying two identical items from the same merchant on the same day). The system assumed it was an OCR error or a duplicate scan, and completely omitted the subsequent identical transactions from being inserted into the database.

**Root Cause 1: Database Deduplication (Save Phase)**
In `discord-bot/lib/credit-card-agent.js`, the `source_ref` (which acts as a unique constraint for the `cc_credit_cards` database table) was strictly composed of:
`cc-dedup-{Bank}-{Date}-{NormalizedName}-{Amount}`

Because identical purchases share all these attributes, any duplicate items within the *same statement* generated the exact same `source_ref`. The batch insert endpoint uses `source_ref` to prevent duplicates across reruns, meaning the second identical transaction in the batch was simply ignored as a "skipped duplicate".

**Root Cause 2: Fallback Consensus Map (Extraction Phase)**
If the models did not achieve a "Perfect Math Override", the fallback consensus grouped items by `date|price`. The `itemMap` explicitly ignored duplicate items from the same model (`if (!entry.models.includes(result.label))`) to prevent an AI hallucination where it repeated the same item. However, this perfectly wiped out *genuine* identical items before they could even reach the save phase.

## 🛠️ The Solution: Relative Occurrence Indexing
To solve this without breaking idempotency, we added occurrence indexing in both places:

**1. Consensus Phase (`itemMap` fix):**
Added an internal model occurrence tracker (`seenByModel`) which appends an index to the anchor key.
`const key = \`baseKey|seenByModel[baseKey]\`;`
This preserves identical transactions across models and perfectly groups `item1` with `item1`, and identical `item2` with `item2`.

**2. Database Save Phase (`source_ref` fix):**
1. A `seenRefs` tracking object is initialized for the batch.
2. For each transaction, a `baseRef` is calculated.
3. If `baseRef` has already been seen in this batch, we append `-2`, `-3`, etc., to create a `finalRef`.
4. This `finalRef` is saved as the `source_ref`.

```javascript
  const seenRefs = {};

  const batch = items.map(item => {
    let finalDate = item.date || new Date().toISOString().split('T')[0];
    const normName = (item.name || '').toLowerCase().replace(/[^a-z0-9]/g, '').slice(0, 15);
    
    let baseRef = `cc-dedup-${bankDisplay}-${finalDate}-${normName}-${Math.abs(Number(item.price)||0)}`;
    seenRefs[baseRef] = (seenRefs[baseRef] || 0) + 1;
    let finalRef = baseRef;
    if (seenRefs[baseRef] > 1) {
      finalRef = `${baseRef}-${seenRefs[baseRef]}`; // Output e.g., cc-dedup-...-100-2
    }
    
    // ...
    return {
      // ...
      source_ref: finalRef,
    };
```

**Why this is robust:**
- Identical purchases on the same statement get unique `source_ref` keys (e.g., `-2`, `-3`), allowing all of them to be safely saved to the database.
- If the *same statement* is scanned again by the cron job, the items will generate the exact same sequence of keys (`baseRef`, `baseRef-2`, `baseRef-3`). The database will recognize these exact keys as existing records and correctly skip them!
