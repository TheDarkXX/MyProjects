## 📋 Compiled Truth & Timeline

**The Problem:** The user noticed that Oracle hasn't produced any findings today. Investigation revealed that the Oracle agent has actually *never* successfully run since its deployment. It was getting stuck in a 15-step `Max steps reached` loop, using 0 tools and generating 0 findings.

**The Root Cause:**
1. **Wrong CLI Flags:** `oracle.js` was using `openclaw agent -m "openai-codex/gpt-5.5" -t '{"messages":...}'`. In OpenClaw CLI, `-m` is `--message` (the prompt text) and `-t` is `--to` (the recipient number). This resulted in the JSON prompt being treated as a recipient number, and the model string being sent as the actual message. GPT-5.5 received a nonsensical prompt and returned `NO_REPLY`.
2. **Gateway Auth Bypass:** `oracle.js` explicitly bypassed the gateway by setting `OPENCLAW_GATEWAY_URL = ''`. However, as established in `V12.3.6`, GPT-5.5 MUST be routed through the OpenClaw Gateway because it relies on ChatGPT web scraping. Bypassing the gateway causes the call to fail.
3. **JSON Array Mismatch:** OpenClaw CLI's `--message` flag expects a plain text string, but the script was passing a stringified JSON array (`[{role: 'system'...}]`).

**The Fix:**
- Updated `oracle.js` to format the `messages` array into a single text string (`[SYSTEM]\n... \n\n [USER]\n...`).
- Corrected the OpenClaw CLI command to use `--model "openai-codex/gpt-5.5" --session-id "oracle-ai" --message '...'`.
- Removed the gateway bypass and correctly passed the token via `OPENCLAW_GATEWAY_TOKEN = process.env.GATEWAY_AUTH_TOKEN`.
- Added robust JSON repair logic (trimming markdown code blocks and trailing commas) to handle cases where the model returns markdown-wrapped JSON.
- Reduced the max agent loop steps from 15 to 8 to prevent excessive timeouts.

3. **OpenClaw CLI ไม่รับ JSON Array:** คำสั่ง `openclaw agent` รับ prompt เป็น plain text ผ่าน `--message` เท่านั้น ไม่สามารถรับ JSON Array `[{role: 'system'...}]` ตรงๆ ได้

## 🛠️ Implementation Steps

### 1. แก้ไข `callGPT55()` ใน `oracle.js`
ปรับปรุงการสร้างคำสั่ง CLI และการจัดการ Environment Variable

```javascript
// [อดีต]
const payload = JSON.stringify({ messages, tool_choice: "none" });
const escapedPayload = payload.replace(/'/g, "'\\''");
const cmd = `openclaw agent -m "openai-codex/gpt-5.5" -t '${escapedPayload}' --timeout 300000 --json`;

// [แก้ไขเป็น]
// 1. Flatten array ข้อความให้กลายเป็น String ก้อนเดียวเพื่อให้ CLI เข้าใจ
const promptText = messages.map(m => `[${m.role.toUpperCase()}]\n${m.content}`).join('\n\n');
const escapedPayload = promptText.replace(/'/g, "'\\''");

// 2. ใช้ flag ที่ถูกต้อง: --model, --session-id, และ --message
const cmd = `openclaw agent --model "openai-codex/gpt-5.5" --session-id "oracle-ai" --message '${escapedPayload}' --timeout 300000 --json`;
```

### 2. แก้ไข Gateway Auth
```javascript
// [อดีต]
envCopy.OPENCLAW_GATEWAY_URL = '';

// [แก้ไขเป็น]
// ส่ง Token ไปแทน เพื่อยืนยันสิทธิ์กับ Gateway
envCopy.OPENCLAW_GATEWAY_TOKEN = process.env.GATEWAY_AUTH_TOKEN;
```

### 3. เพิ่มการทำความสะอาด (Repair) JSON Response
เนื่องจากเราส่ง prompt ในรูปแบบ text, โมเดลอาจจะติดนิสัยพิมพ์ markdown ````json ... ```` กลับมา หรือมีข้อความนำหน้า/ตามหลัง
```javascript
// [แก้ไขเป็น]
let cleanText = replyText.replace(/^```json\s*/gm, '').replace(/^```\s*/gm, '').trim();
// ลบ comma เกิน (trailing comma) เผื่อโมเดลพิมพ์ผิด
cleanText = cleanText.replace(/,\s*([}\]])/g, '$1');
// ดึงออกมาเฉพาะส่วนที่เป็น JSON
const jsonMatch = cleanText.match(/\{[\s\S]*\}/);
if (jsonMatch) cleanText = jsonMatch[0];

parsed = JSON.parse(cleanText);
```

### 4. ลด Max Steps (จาก 15 เหลือ 8)
เพื่อป้องกันไม่ให้ Script รันค้างนานเกินไป (15 steps อาจกินเวลาเป็นชั่วโมง) 8 steps เพียงพอสำหรับ Oracle ในการดึงข้อมูล, วิเคราะห์, และสรุปผลแล้ว
