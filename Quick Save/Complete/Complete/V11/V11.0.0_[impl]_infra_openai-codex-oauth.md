# 🚀 OpenAI Codex OAuth Integration (GPT-5.4)

## 📌 Context
User wanted to integrate OpenAI GPT-5.4 and GPT-5.5 into the existing AI Gateway using their ChatGPT Plus subscription via the **Codex OAuth flow**. This allows free usage of flagship models for the OpenClaw agents, bypassing standard API pay-as-you-go costs.

## 🛠️ What Was Done
1. **Codex CLI Installed:** Installed `@openai/codex` CLI on the VPS.
2. **Device Auth Flow:** Generated a Device Auth code, user authenticated via browser, generating `~/.codex/auth.json`.
3. **AI Gateway Updated (`ai-gateway.js`):**
   - Added `/openai/` route forwarding to `api.openai.com`.
   - Added `getCodexToken()` to extract `tokens.access_token` from `auth.json`.
   - Injected `Authorization: Bearer <token>` into OpenAI requests.
   - Added 401 interceptor: Automatically runs `codex models` via `execSync` to trigger a token refresh when an unauthorized response is received.
4. **OpenClaw Config Updated (`openclaw.json`):**
   - Added `openai-codex` provider pointing to `http://127.0.0.1:18810/openai/v1`.
   - Added models: `gpt-5.4`, `gpt-5.5`.
   - Updated `coding` agent to use `openai-codex/gpt-5.4` as the primary model.

## 🧠 Decision & Rationale
- **Why Codex OAuth over API Key?** User has a ChatGPT Plus subscription and wanted to save costs.
- **Why GPT-5.4 for "คุยเล่น"?** GPT-5.4 was chosen over 5.4-mini (too simple) and 5.5 (too heavy) as the best balance of speed, intelligence, and large context capacity for retrieving Brain/RAG data fluently.
- **Why trigger `codex models` on 401?** To automate token refreshes natively utilizing the Codex CLI's built-in refresh capability without needing to reverse engineer the OAuth PKCE flow.

## Architecture: Codex OAuth Flow

```
┌───────────────┐     OAuth Login      ┌──────────────┐
│  codex login   │ ─────────────────→  │ OpenAI Auth   │
│  (บน VPS)      │ ←── access_token ── │ Server        │
└───────┬───────┘    + refresh_token    └──────────────┘
        │
        │ เก็บใน ~/.codex/auth.json
        ▼
┌───────────────────────────────────────────────────────┐
│              ai-gateway.js (port 18810)               │
│                                                       │
│  /openai/*  →  อ่าน token จาก auth.json               │
│              →  inject Authorization: Bearer <token>   │
│              →  forward ไป api.openai.com              │
│              →  ถ้า 401 → auto refresh token           │
│              →  log tokens ลง usage.json               │
└───────────────────────────────────────────────────────┘
```

### Token Lifecycle

| เรื่อง | รายละเอียด |
|--------|------------|
| **access_token** | หมดอายุบ่อย (ประมาณ 1-2 ชม.) |
| **refresh_token** | ใช้ auto-renew ได้ ถ้า auth.json ถูกจัดการโดย process เดียว |
| **Auto-refresh** | Gateway ดัก 401 → ใช้ refresh_token ขอ access_token ใหม่ |
| **Fallback** | ถ้า refresh ล้มเหลว → ต้อง re-login ด้วย `codex login` ใหม่ |

### ทำไมน่าจะเสถียร

1. **OpenClaw** ใช้ Codex OAuth เป็น first-class provider (มี `openclaw onboard --auth-choice openai-codex`)
2. **Hermes** มี credential pool + atomic refresh ที่ handle token rotation อัตโนมัติ
3. **Codex CLI** มี built-in auto-refresh mechanism + device-auth สำหรับ headless server
4. OpenAI ยังไม่ได้ block flow นี้ เพราะ ecosystem ใหญ่เกินไป

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Token expire + refresh fail | Medium | Agent ไม่มี GPT-5.x | Watchdog alert + manual re-login |
| OpenAI block OAuth | Low (short-term) | GPT-5.x ใช้ไม่ได้ | Fallback to API Key |
| Rate limit ของ Plus | Medium | Slow responses | ไม่ใช้กับ high-frequency cron |
| Token race condition | Low | Auth error | Single-process token management |

> [!TIP]
> **Cost savings:** ถ้าใช้ Codex OAuth สำเร็จ — GPT-5.4 ฟรี (รวมใน Plus $20/เดือน) แทนที่จะจ่าย ~$2.50/M input + $15/M output ผ่าน API
