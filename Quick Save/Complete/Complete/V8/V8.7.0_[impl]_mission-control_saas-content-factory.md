# Mission Control SaaS — Architecture & Implementation

## 🧠 Core Decisions & Journey

The user requested a massive expansion of the OpenClaw system: a new "Mission Control" module acting as a scalable Content Factory. 
Initially, the prompt considered building this as a standalone mini-app/website. However, following the architectural review, we decided to **embed it within the existing Brain App SPA** as a new sidebar category. This saved immense operational overhead (Auth, API, DB connections, PM2 setup).

### Phases Completed:
1. **Foundation:** Created mc_projects, mc_templates, mc_jobs, and mc_analytics in SQLite. Built the UI tab using the strict BillPay Dark Dashboard design system.
2. **Dynamic UI:** Created a fully schema-driven form renderer that adapts to template JSON schemas.
3. **Pipeline (Worker):** Built mc-worker.js to run every 10 seconds. It processes pending jobs, auto-approves them (if project settings allow), and pushes them to b_post_queue.
4. **AI Generation Loop:** Implemented the "Value in Comments" template. The worker intercepts i-text-gen jobs, calls i-gateway.js using gemini-2.5-flash, and formats the response directly into clickbait headlines and step-by-step comment arrays.
5. **Analytics Feedback Loop:** Modified b_post_queue to track mc_job_id. Updated the Facebook cron script to write native b_post_id back to Mission Control upon successful publishing. Finally, added an hourly Analytics sync loop in mc-worker.js that hits the Graph API to fetch live views, shares, and comments for posts made in the last 7 days.

### Phase 3: Worker & Pipeline Integration (Current Phase)

> [!IMPORTANT]
> **User Review Required:** This phase introduces the background processing engine that will convert Mission Control jobs into actual Facebook posts. Please review the architecture below before we implement.

**1. Mission Control Worker (`routes/mc-worker.js`)**
We will create a background loop (running every 10-30 seconds via `setInterval` in `server.js`) that acts as the factory assembly line:

* **Step A: Auto-Approval Routing**
  * Scan `mc_jobs` where `status = 'pending'`.
  * For basic templates where inputs are already provided manually via the form, we bypass the AI generation for now.
  * Update `status = 'completed'`.
  * If the project's `approval_mode` is `auto`, automatically set `approval_status = 'approved'`. Otherwise, leave as `pending` for the user to approve via Gallery.

* **Step B: Publishing Queue Sync (`mc_jobs` → `fb_post_queue`)**
  * Scan `mc_jobs` where `approval_status = 'approved'` and `publish_status = 'unpublished'`.
  * Insert a new record into `fb_post_queue` mapping the fields:
    * `message` ← `input_variables.message`
    * `bg_preset_id` ← `input_variables.bg_color`
    * `media_url` ← `input_assets.image` or `input_assets.video`
    * `comments` ← `input_variables.comments` or `[input_variables.first_comment]`
    * `scheduled_by` ← `'mission-control'`
  * Update `mc_jobs.publish_status = 'queued'`.

**2. API Endpoints Update (`routes/mission-control.js`)**
* Add a `POST /jobs/:id/approve` endpoint so users can manually approve jobs from the Gallery UI.
* Add a `POST /jobs/:id/reject` endpoint.

**3. Frontend Updates (`mission-control.js`)**
* Add "Approve" / "Reject" buttons to the Gallery cards for jobs that are `pending` approval.
* Add auto-refresh polling (every 10s) on the Gallery/Dashboard so the user sees jobs moving from `pending` → `approved` → `queued` → `posted` in real-time.
