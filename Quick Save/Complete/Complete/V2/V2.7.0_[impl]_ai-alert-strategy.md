# AI Alert & Fallback Strategy (V2.7.0)

เพื่อป้องกันปัญหา AI ตอบไม่ได้และไม่มีคนสังเกตเห็น เราจะใช้ข้อมูลจาก `ai-status.json` ที่เราเพิ่งทำ มาเป็นตัวตัดสินใจในการยิงแจ้งเตือนและสลับโมเดลอัตโนมัติ (Fallback)

## User Review Required
> [!IMPORTANT]
> กรุณารีวิวเงื่อนไขการแจ้งเตือนด้านล่าง ว่าต้องการให้แจ้งเตือนเข้าช่องทางไหน (เช่น Discord ห้อง Alerts หรือ Notify ไปมือถือ) และระยะเวลาการรอ (Threshold) สมเหตุสมผลหรือไม่ครับ

## 🚨 Scenarios & Alert Triggers (เงื่อนไขการแจ้งเตือน)

การตั้งเงื่อนไขที่ดีต้อง **ไม่แจ้งเตือนพร่ำเพรื่อ (No Spam)** ถ้าเน็ตเวิร์คแค่กระตุก 1 ครั้งจะไม่เตือน แต่จะเตือนเมื่อพังจริงๆ เท่านั้น

1. **[CRITICAL] OpenClaw Gateway Down (เชื่อมต่อ Gateway ไม่ได้เลย)**
   - **เงื่อนไข:** `ECONNREFUSED` หรือโมเดลทั้ง 4 ตัว `status = offline` พร้อมกันทั้งหมด
   - **แจ้งเตือน:** 🔴 "CRITICAL: OpenClaw Gateway Offline! All AI Agents are paralyzed." (แจ้งทันทีไม่ต้องรอ)
   - **Action:** ส่ง Notify เข้ามือถือ + Discord ทันที

2. **[WARNING] Model Offline / Timeout (โมเดลตัวใดตัวหนึ่งพังต่อเนื่อง)**
   - **เงื่อนไข:** โมเดลสถานะ Failed หรือ Timeout **ติดต่อกัน 3 ครั้ง (15 นาที)** (ป้องกันเคส API กระตุกชั่วคราว)
   - **แจ้งเตือน:** 🟡 "WARNING: [Model] is offline for 15 minutes."
   - **Action:** ระบบจะทำเครื่องหมายโมเดลนั้นว่า Dead ชั่วคราว

3. **[WARNING] High Latency (เริ่มช้าผิดปกติจนอาจทำให้ Bot Timeout)**
   - **เงื่อนไข:** Latency สูงกว่า `20,000ms` (20 วิ) **ติดต่อกัน 3 ครั้ง**
   - **แจ้งเตือน:** 🟠 "DEGRADED: [Model] is experiencing high latency (>20s)."

4. **[RESOLVED] System Recovered (ระบบกลับมาปกติ)**
   - **เงื่อนไข:** โมเดลที่เคยแจ้งเตือนว่าพัง กลับมาเช็คผ่าน (OK) **ติดต่อกัน 2 ครั้ง (10 นาที)**
   - **แจ้งเตือน:** 🟢 "RESOLVED: [Model] is back online and stable."

## 🔄 Automated AI Fallback Strategy (แผนสลับ AI อัตโนมัติ)

สคริปต์ต่างๆ (เช่น `cron-gmail.js`, `health-news.js`) ก่อนจะยิง API ควรอ่านค่าจาก `ai-status.json` ก่อนเสมอ เพื่อหาโมเดลที่แข็งแรงที่สุดตามลำดับ (Tier)

- **Tier 1 (ตัวหลัก):** `AutCodex: GPT 5.4` (ฉลาดสุด)
- **Tier 2 (สำรอง 1):** `AutCodex: GPT 5.1` (ฉลาดรองลงมา, เสถียรและเร็วกว่า)
- **Tier 3 (สำรอง 2):** `Kie: Gemini 2.5 Pro` (ข้าม Provider กรณี OpenClaw มีปัญหาด้านวงเงินหรือ API ของฝั่ง OpenAI ล่ม)
- **Tier 4 (ไม้ตายสุดท้าย):** `Kie: Gemini 2.5 Flash` (เร็วจี๋ โอกาสล่มยากที่สุด เอาไว้ใช้อ่าน/สรุปคร่าวๆ เพื่อให้งานไม่สะดุด)

**Flow การทำงานที่วางแผนไว้:**
เมื่อ Bot เริ่มทำงาน -> โหลด `ai-status.json` -> เช็คว่า `Tier 1` status == 'online' และ latency < 20000ms หรือไม่ -> ถ้าพัง ให้ไหลลงไปหา `Tier 2` -> ถ้าพังอีก ไหลไป `Tier 3` ตามลำดับ
