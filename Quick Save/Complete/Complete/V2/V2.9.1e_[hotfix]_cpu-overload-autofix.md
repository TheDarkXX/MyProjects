# V2.9.1 CPU Overload Autofix & UI Optimization

## 📖 Context & Problem (ที่มาและปัญหา)
- **ปัญหาเดิม**: ระบบแสดงกราฟ CPU โหลดแค่ในส่วนของ UI Dashboard (ในรูปแบบ Load Average) รวมถึงสคริปต์ `health-check.sh` แบบเดิมแจ้งเตือนเฉพาะเวลาที่ RAM/Disk ใกล้เต็ม แต่ **ไม่มีหลอดวัดไฟของ CPU Overload**
- **ผลกระทบ**: ถ้าเซิร์ฟเวอร์โดนบอทสแปม หรือรันโพรเซส AI ที่ลูปค้าง (Hanging) จน CPU แช่ 100% นานๆ เซิร์ฟเวอร์จะค้าง (Timeout) และไม่สามารถกอบกู้ตัวเองได้ 
- **โจทย์**: วางระบบ Monitor + Alert (เตือนสาเหตุ/Procs ที่กิน) + Auto-fix (แก้แฮงค์) พร้อมปกป้องระบบไม่ให้ตัว Auto-fix เผลอมั่วไป Kill โพรเซสที่สำคัญ

## 🛡️ Implementation Strategy (แนวคิดและมาตรการแก้ปัญหา)
ผมได้รื้อสคริปต์ `health-check.sh` บางส่วนและเพิ่ม **Load Average (5 นาที)** เข้าไปผูกกับการทำงานแบบ **Tiered Escalation** (แบ่งเป็นลำดับชั้นความรุนแรง) เพื่อเป็น Best Practice ดังนี้:

1. **ใช้ 5-Min Load Average**: ลดการแจ้งเตือนผิดพลาดเวลาเกิด Peak แว้บเดียว (ไม่ลั่นไฟเหลืองง่ายๆ)
2. **ระบบนับ Streak (Hit count)**: Cron ทำงานทุกๆ 15 นาที ระบบจะนับว่า CPU ค้างเกินเกณฑ์มาแล้วกี่รอบ (เพื่อให้แน่ใจว่ามันค้างจริง ไม่ได้กำลังทำงานปกติ)
3. **กำหนดชั้นความรุนแรง 4 ระดับ**: 
   - **Yellow (1 hit)**: เก็บข้อมูลและเตือน แต่เปิดทางระบบทำงานต่อ
   - **Level 1 (2 hits, 30 นาที)**: Restart เฉพาะเซอร์วิสเบาๆ (`brain-app`, `discord-bot`)
   - **Level 2 (2 hits, ทะลุเกณฑ์ 200%)**: สั่ง Restart All ยกระดับการป้องกันทันที
   - **Level 3 (3 hits, ทะลุเกณฑ์ 300%)**: ค้างนาน 45 นาทีแปลว่า OS แฮงค์ จึงสั่งกระบวนท่า `/sbin/reboot`
4. **Safety Net (Best Practice)**:
   - ห้ามใช้ `pm2 reload all` เพราะเกิด Process spawn เบิ้ลในภาวะโหลดหนัก (เปลี่ยนไปใช้ `restart` แทน)
   - ห้ามสั่ง `drop_caches` พร่ำเพรื่อในภาวะ I/O หนัก
   - 📌 **Cooldown Rule**: เพิ่มตั้งเวลา cooldown 5 นาที หลังการ autofix ทุกครั้ง ไม่ให้มัน restart ซ้อนกันรัวๆ จนบูตไม่ขึ้น
