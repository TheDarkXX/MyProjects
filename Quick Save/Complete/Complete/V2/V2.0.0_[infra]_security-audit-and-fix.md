## 📖 เหตุผล และที่มาที่ไป (Background & Rationale)

**1. ปัญหานี้เกิดขึ้นได้อย่างไร?**
- โดยปกติในจังหวะขึ้นโครงระบบหรือทำ Prototype เรามักจะเอา API Keys หรือ Tokens แปะลงไปในโค้ดตรงๆ (Hardcoded) เพื่อให้รันง่ายและเร็ว เช่น พิมพ์ลงไปใน `openclaw.json` หรือเขียนแปะไว้ในไฟล์โน้ต `.md`
- แต่เนื่องจากไฟล์เหล่านี้ถูกจัดการด้วย Git เมื่อเราสั่ง Commit และ Push (`git push vps`, `git push origin`) Key เหล่านั้นก็ถูกส่งขึ้นไปเก็บถาวรบนเซิร์ฟเวอร์ VPS และ GitHub ด้วย

**2. ทำไมถึงเป็นเรื่องคอขาดบาดตาย? (ความเสี่ยง)**
- **ความลับใน Git History อยู่ยงคงกระพัน:** กฎเหล็กของ Git คือ "ลบของในไฟล์ ไม่ได้แปลว่าลบจากประวัติ" ถึงวันนี้เราจะลบ Key อออกจากโค้ด แล้ว commit ใหม่ คนที่เข้าถึง Repo ก็ยังสามารถย้อนประวัติ (Git Log/History) ไปดูเวอร์ชันเก่าเพื่อกู้ Key ออกมาได้อยู่ดี
- **Private Repo ก็ไม่ปลอดภัย 100%:** แม้ว่าโปรเจ็กต์จะตั้งเป็น Private แต่ถ้าแอคเคาท์ GitHub ถูกแฮ็ก, Token การเข้าสู่ระบบหลุดรอด, หรือเครื่องคอมพิวเตอร์นี้โดนสปายแวร์ ใครก็ตามที่เข้าถึง Source Code นี้ได้ จะได้กุญแจ "ผี" ที่ไขเข้าระบบ Discord, Telegram, โมเดล AI (เสียเงิน), รวมถึง Gateway ของหลังบ้านเราได้ทั้งหมดทันที

**3. วิธีที่ถูกต้อง (Best Practice) ที่เรากำลังจะทำ**
- ปรัชญาความปลอดภัยคือ **ต้องแยก "Code" กับ "Secrets" ออกจากกันอย่างเด็ดขาด**
- ข้อมูล Code: วางบน GitHub, แจกจ่ายให้ทีมงาน, หรือเปิด Public ได้ 
- ข้อมูล Secrets: ต้องไม่ปรากฏใน Code เลยแม้แต่บรรทัดเดียว แต่ต้องถูกเก็บแยกในแฟ้มเฉพาะ (เช่น `.env`) และอยู่ในไฟล์งดเว้น (`.gitignore`) เพื่อให้ Git มองไม่เห็นแฟ้มนี้
