# 🚀 Implementation Plan: Hydra Dedup Status Filter Hotfix

## 📌 Context (Compiled Truth)

### Context & Root Cause
The user noticed that even after the deduplication fixes applied this morning (V10.1.0) and yesterday, Ideator was STILL generating duplicate proposals for the exact same PM2 crash loops and stuck content ideas. 

**Why was this happening?**
1. **The SQL Status Blindspot:** In both `inspector.js` and `ideator.js`, the SQL deduplication query was explicitly filtering: `AND status IN ('proposed', 'approved', 'awaiting_human', 'deployed', 'shipped')`. 
2. **The Reject/Icebox Loop:** When we use the `/evo` skill to reject or icebox a proposal, its status changes to `rejected` or `iceboxed`. Because those statuses were omitted from the `IN (...)` clause, the database query returned `null`.
3. **The Result:** Inspector and Ideator "forgot" that the proposal ever existed the moment we rejected it, immediately recreating it on the next run.

## 🛠️ Proposed Changes

### 1. Fix Inspector Deduplication (`scripts/hydra/inspector.js`)
- **Action:** Removed the `AND status IN (...)` filter from the `recentEvo` query entirely. Now, if *any* evolution_log entry exists for the `dedup_key` within the last 14 days (regardless of whether it was rejected, iceboxed, or failed), it will correctly suppress new findings.

### 2. Fix Ideator Deduplication (`scripts/hydra/ideator.js`)
- **Action:** Removed the `AND status IN (...)` filter from the `activeSameCategory` query entirely, applying the identical 14-day absolute block.

## 💻 Code Snippets

### Before (Flawed):
```javascript
// Inspector & Ideator had variations of this:
const recentEvo = db.prepare(`
  SELECT id, title, status FROM evolution_log
  WHERE dedup_key = ? AND status IN ('proposed', 'approved', 'awaiting_human', 'deployed', 'shipped') 
  AND created_at > datetime('now', '-14 days') LIMIT 1
`).get(findingDedupKey);
```

### After (Fixed):
```javascript
// Removed the status IN filter completely
const activeSameCategory = db.prepare(`
  SELECT id, title FROM evolution_log 
  WHERE dedup_key = ?
  AND created_at > datetime('now', '-14 days')
  LIMIT 1
`).get(findingDedupKey);
```
