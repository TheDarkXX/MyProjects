# V10.4.1 Credit Tracker — Triple Fix

## 📦 RAW ARTIFACT BACKUP
_No standalone artifacts were created. All findings are from direct code audit and VPS log analysis._

## 🐛 Bug 1: Wrong Field for Skip-Logic (statement_month vs created_at)

**Root Cause:** `checkImportedBanksCount()` in `cron-cc-statements.js` checked `item.created_at.startsWith(currentYearMonth)` to determine if the current month's bills had already been imported. But bills for March 2026 were imported on 2026-04-12, causing the system to see 4 banks with `created_at` in `2026-04`, falsely concluding April was complete.

**Fix:** Changed to check `item.statement_month === currentYearMonth` instead.

```diff
- if (item.created_at && item.created_at.startsWith(currentYearMonth) && item.credit_card) {
-   uniqueBanks.add(item.credit_card.toUpperCase());
- }
+ if (item.statement_month === currentYearMonth && item.credit_card) {
+   uniqueBanks.add(item.credit_card.toUpperCase());
+ }
```

## 🐛 Bug 2: Missing Crontab Entry

**Root Cause:** The crontab on VPS did not contain any entry for `cron-cc-statements.js`. Per V2.2.1, the schedule should be `30 8 15-30 * *`. It was likely lost during a crontab edit or server migration.

**Fix:** Re-added the crontab entry:
```
30 8 15-28 * * cd /root/brain-app/discord-bot && /usr/bin/node scripts/cron-cc-statements.js >> /tmp/cron-cc-statements.log 2>&1
```

**Note:** Changed `15-30` to `15-28` because months can have at most 28-31 days, and cron handles day-of-month correctly, but 28 is a safe universal cap.

## 🐛 Bug 3: Deprecated Backup Model (OpenRouter 404)

**Root Cause:** The backup model `qwen/qwen2.5-vl-32b-instruct` was removed from OpenRouter and returns HTTP 404. This caused all backup verifications to fail silently, leaving any math-mismatched statements permanently in HELD state.

**Fix:** Replaced with `qwen/qwen3-vl-32b-instruct` which is the current equivalent on OpenRouter.

```diff
- const backupModel = { id: 'qwen/qwen2.5-vl-32b-instruct', provider: 'openrouter' };
+ const backupModel = { id: 'qwen/qwen3-vl-32b-instruct', provider: 'openrouter' };
```

## Changelog: Phase 2 (Force-Save & Model Upgrade)
- **Model Upgrade**: Replaced 32B backup with `qwen/qwen3-vl-235b-a22b-instruct` for superior math reasoning on edge cases.
- **Prompt Engineering**: Explicitly instructed the AI to extract ONLY "Total New Charges" (ยอดรวมรายการเรียกเก็บใหม่) and ignore Previous Balances to prevent math mismatches on SCB/CardX.
- **Force Save Mismatches**: Modified `shouldSave` logic to bypass the `HELD` status for `MATH_MISMATCH` and `MISMATCH`, saving them to `cc_credit_cards` with a warning flag, allowing manual review on the Web UI instead of leaving them completely unprocessed.
- **Gmail API Fix**: Added optional chaining to `msgData.payload?.headers` to prevent crashes when Gmail returns malformed API responses.
- **Storage Restructure**: 
  - Changed PDF filename format from `[bankKey]_[date]_[hash].pdf` to `[BankDisplay]-[YYYY-MM-DD].pdf` (e.g., `SCB-2026-04-27.pdf`).
  - Moved the storage path from `discord-bot/data/cc-statements` directly to `Openclaw-VPS/cc-statements` at the project root for easier local access.

## Changelog: Phase 3 (Prompt Hardening & Piggyback Sync)
- **Piggyback PDF Sync**: Modified the local `.git/hooks/pre-push` to run `scp -r root@185.250.38.247:/root/brain-app/cc-statements` in a hidden background PowerShell window. This silently syncs new PDFs to the local PC whenever the user runs `git push vps` (zero manual effort, zero resource bloat).
- **Discord Notification Fix**: Fixed a bug where the `details` array in `buildConsensus` hardcoded only the primary model, preventing the backup model (235B) from rendering in the Discord mismatch log.
- **Krungsri Prompt Upgrade**: Explicitly instructed the model to ignore the entire "รายการชำระเงินสำหรับบัตรเครดิต" (Payment Transactions) section to stop it from extracting negative cash payments as refunds. Fixed installment targeting.
- **SCB Prompt Upgrade**: Added strict guardrails to ignore "PREVIOUS BALANCE" and "Payment received". Also blocked the model from using "TOTAL BALANCE" as the statement_total if there is a previous balance, preventing massive math mismatches.
- **Infinite Loop Fix (BBL 0-Items)**: If a bank has 0 transactions (like an unused BBL card), it previously triggered a `SUSPICIOUS` failure, preventing DB insertion. Because it wasn't inserted, `cron-cc-statements.js` never counted it as completed, causing an infinite cron loop every single day from the 15th to 28th. Fixed by identifying genuine empty statements and injecting a dummy "No Activity" row with 0 price. This successfully saves to the DB and breaks the loop.

## Changelog: Phase 4 (ReferenceError Fix)
- **Discord Notification Crash Fix**: The Phase 3 fix for Discord notifications introduced a logic bug where `backupResult` was declared as a block-scoped constant (`const backupResult`) inside the backup step, but was then referenced globally during the `details` array construction (`buildConsensus` payload). This caused a `ReferenceError` when the backup model was triggered. Fixed by properly declaring `let backupResult = null;` in the outer scope before model evaluation.
