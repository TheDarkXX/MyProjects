# 🔬 [hotfix] Hydra Ideator Dedup Root Cause Fix

## 📌 Context (Compiled Truth)
Ideator has been repeatedly proposing optimizations that already exist in the codebase (e.g., L1 LRU Caches, fire-and-forget writes). Previous attempts to fix this by adding `rejection-patterns` and `CORE_INFRA_FILES` context injections failed.

A deep root cause analysis revealed 5 stacked failures:
1. **UTF-16 BOM Corruption:** PowerShell `echo >>` commands from the AG `/evo` skill wrote UTF-16 lines into the JSONL memory files. The `readJSONL()` function silently dropped these lines. All rejection rules generated since April 26 were invisible to the system.
2. **Pre-filter Logic Flaw:** The memory pre-filter was comparing `proposalData.category` against the rejected `category` (e.g. `experiment` !== `dedup_failure`), ignoring the actual rejection pattern phrase.
3. **LLM Inattention:** The `minimax-m2.7` model failed to reliably cross-reference the 8000-char injected context against its own creative ideas.
4. **Missing Post-Gen Verification:** There was no deterministic codebase check after the LLM generated a proposal.
5. **Unprotected Innovation Mode:** The `bold` and `moonshot` personas bypassed ALL dedup gates, which is where the duplicate `experiment` proposals originated.

### What Was Done
1. **Scrubbed Corrupted Data:** Built `clean-jsonl.cjs` to purge null bytes. Restored 34 valid feedback lines and 7 valid rejection patterns.
2. **Hardened the Policy:** Updated `/evo` skill manual to strictly forbid PowerShell `echo >>`, enforcing Node `write_to_file` (UTF-8).
3. **Fixed Pre-Filter:** Updated `ideator.js` to substring match `p.pattern` against `proposalData.title`.
4. **Added The Iron Gate:** Added a post-generation deduplication scanner in both the pool path AND the Innovation Mode path. It reads the target file and if >70% of the proposed action keywords already exist in the file, it drops the proposal immediately.

# 🔬 Root Cause Analysis: Why Ideator Dedup Is Still Broken

## TL;DR — It's not one bug, it's 5 structural failures stacked on top of each other

Every "fix" so far has patched a **symptom** while the root causes remain intact. Here's the full chain:

---

## Root Cause #1: 🔴 Corrupted Memory Files (UTF-16 BOM Poison)

**This is the #1 killer.** Multiple memory files are silently corrupted with UTF-16LE encoding (null bytes between every character):

| File | Lines Affected | Impact |
|------|---------------|--------|
| `feedback-log.jsonl` | Lines 17-35 (latest entries) | ❌ **Cannot be parsed as JSON** |
| `rejection-patterns.jsonl` | Lines 3-4, 7-8 | ❌ **Pre-filter cannot read these patterns** |

**What happens in code:**

```javascript
// hydra-memory.js line 322-327
function readJSONL(filePath) {
  const lines = fs.readFileSync(filePath, 'utf-8').trim().split('\n').filter(Boolean);
  return lines.map(line => {
    try { return JSON.parse(line); } catch { return null; }  // ← SILENTLY swallows the error
  }).filter(Boolean);  // ← Corrupted lines become null → filtered out → GONE
}
```

The `readJSONL` function **silently drops** any line it can't parse. So all your carefully-written rejection patterns from April 26 onwards are **invisible to the Ideator**.

**Why it's corrupted:** The `echo '...' >> file.jsonl` command in PowerShell writes UTF-16LE by default. Lines 1-2 and 5-6 in `rejection-patterns.jsonl` are clean because they were written by Node.js (UTF-8). Lines 3-4 and 7 were written by PowerShell `echo` commands from AG sessions → UTF-16 → corrupted.

> **This is why "fixing" keeps failing** — every time we append a new rejection pattern via PowerShell, it goes in as UTF-16 garbage that `JSON.parse` can't read.

---

## Root Cause #2: 🟠 Pre-filter Matches on `category`, Not on `pattern` Content

Even if the rejection patterns *were* readable, the pre-filter logic has a fundamental design flaw:

```javascript
// ideator.js line 448-450
const matchingPattern = rejectionPatterns.find(
  p => p.category === proposalData.category && (p.rejectRate || 0) >= 0.8
);
```

This checks: *"Is this proposal's **category** the same as a previously rejected category?"*

But the duplicate cache proposals (EVO-339, EVO-669) have:
- **category:** `experiment`

And the rejection pattern we wrote has:
- **category:** `dedup_failure`

**They don't match!** The pre-filter is looking for `experiment === dedup_failure` → false → proposal goes through.

The pre-filter was designed to block categories like `"misaligned"` or `"noise"` — not to detect *"this specific optimization already exists in the code"*. It's the wrong tool for the job.

---

## Root Cause #3: 🟠 Code Injection Is Present But Insufficient

Credit where it's due — there IS a `CORE_INFRA_FILES` mechanism (lines 279-283):

```javascript
const CORE_INFRA_FILES = [
  'lib/ai-call.js',        // AI gateway wrapper, circuit breaker, Redis cache
  'lib/heartbeat.js',      // Agent lifecycle, context injection
  'lib/hydra-memory.js',   // Cognitive memory, feedback, blocking
];
```

This reads `ai-call.js` and injects the first **8000 characters** into the LLM context with a warning:

```
--- EXISTING IMPLEMENTATION (lib/ai-call.js) — อ่านให้จบก่อนเสนออะไร 
ถ้าฟีเจอร์ที่จะเสนอมีอยู่แล้วในโค้ดนี้ ห้ามเสนอซ้ำ ---
```

**But the problem is:** `ai-call.js` is 294 lines / ~11KB. The injection only takes `code.slice(0, 8000)` — about **~230 lines**. The L1InMemoryCache class starts at line 72 and the fire-and-forget pattern is at line 210-225.

The 8000-char limit **does** include the L1 cache class (lines 72-130 fit within 8000 chars). But the LLM still proposes it anyway because:

1. The LLM (minimax-m2.7 / GLM-4.7) isn't strong enough to reliably cross-reference "does this already exist in the injected code?" while simultaneously generating creative proposals
2. The instruction `ห้ามเสนอซ้ำ` is buried in a context block among 20+ other injections (skills, plans, memory, etc.)

---

## Root Cause #4: 🟡 Dedup Key Is Based on Finding Category, Not Proposal Content

The database dedup check (line 417-428):

```javascript
const findingDedupKey = affectedComp ? `${item.category}:${affectedComp}` : item.title;

const activeSameCategory = db.prepare(`
  SELECT id, title FROM evolution_log 
  WHERE dedup_key = ?
  AND created_at > datetime('now', '-14 days')
  LIMIT 1
`).get(findingDedupKey);
```

This only prevents the **same finding** from producing multiple proposals. It does NOT check if a **semantically similar proposal** already exists. Two different Inspector findings about `ai-call.js` will produce different dedup keys — and both will pass through even if they propose the same optimization.

---

## Root Cause #5: 🟡 No Post-Generation Verification Step

The entire pipeline is: `Inspector Finding → Ideator LLM → Save to DB → Write Piggyback → Done`

There is **zero post-generation verification**. No step asks: *"Does this proposal describe something that already exists in the codebase?"*

The system trusts the LLM's output completely. Given that the LLM is a mid-tier model (minimax-m2.7, GLM-4.7) with a long context window full of injected code, plans, and memory — it simply doesn't reliably catch that a feature already exists.
