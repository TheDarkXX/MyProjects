# Hotfix: Missing Token Analytics Data (Deploy Hook Overwrite)

## 📌 Context
The USER reported that the Token Analytics Graph on the OpenClaw Dashboard was entirely blank for past days (e.g., April 3 - April 25), with the only visible data plotted on the very latest day. The 30-Day Total (140.2K) was exactly 7x the 7-Day Avg (20.0K), proving that only ~7 days of data existed on the VPS.

## 🚀 Root Cause: Deploy Hook `git checkout -f` Overwrites Production Data

**The kill chain:**
1. `discord-bot/data/` was added to `.gitignore` and `git rm --cached` ran **locally** (commit `068d93b`)
2. On the **VPS**, the working tree still had these files tracked — `.gitignore` does NOT untrack already-tracked files
3. Every `git push vps` → deploy hook ran `GIT_WORK_TREE=/root/brain-app git checkout -f` → **overwrote/deleted** live `usage.json`
4. All accumulated production telemetry data wiped silently

**Evidence:**
- `stat` showed `usage.json` Birth = Apr 23 (matching a deploy)
- `git ls-files` on VPS showed `discord-bot/data/usage.json` STILL TRACKED
- `usage.json.backup` on VPS: only Apr 25-26 (partial recovery attempt)
- Git commit `0b6622c`: Apr 3-19 data (55 sessions)

## 🛠️ 4-Layer Defense Implementation

### Layer 1: Untrack on VPS (Immediate)
```bash
cd /root/brain-app
git rm --cached -r discord-bot/data/
```

### Layer 2: Deploy Hook Protection (Permanent)
Rewrote `post-receive` hook to backup/restore ALL runtime data directories:
- `db/` (SQLite DB) — was already protected
- `discord-bot/data/` — **NEW** (usage.json, expenses, cc-statements)
- `data/` — **NEW** (hydra memory, buffer, strategy)
- `public/data/` — **NEW** (dashboard JSON outputs)

Pattern: backup to `/tmp/*-pre-deploy` before `git checkout -f`, restore after.

### Layer 3: Enhanced Daily Backup (`scripts/daily-backup.sh`)
Extended existing DB-only backup to also cover:
- `discord-bot/data/` → `/root/backups/runtime-data/discord-bot-data_YYYYMMDD_HHMMSS`
- `data/` → `/root/backups/runtime-data/hydra-data_YYYYMMDD_HHMMSS`
- 7-day rotation
- Automatic corruption detection + restore from latest backup

### Layer 4: Data Integrity Watchdog (`scripts/data-integrity-watchdog.cjs`)
Cron runs every 6 hours (`0 0,6,12,18 * * *`). Checks:
- File existence (all critical runtime files)
- Minimum file size (catches empty/truncated files)
- Staleness (>48h without modification)
- JSON validity + session count for usage.json
- Deploy hook integrity (verifies backup sections exist)

### Layer 5: Offsite GitHub Time Machine (`scripts/snapshot-backup.sh` & `scripts/restore-snapshot.sh`)
Ultimate defense against complete VPS failure.
- Created local bare repo on VPS at `/root/openclaw-snapshots` connected to private GitHub remote `myclaw-snapshots`.
- Cron runs daily at `04:00`.
- Script copies DB, discord-bot data, hydra data, public data, and configs.
- Commits as a single snapshot `snapshot: YYYY-MM-DD` and auto-pushes to GitHub.
- Included `restore-snapshot.sh` which allows previewing and rolling back the entire server state to any specific day in the past.

### Bonus: Atomic Writes
Added `tmp` + `renameSync` pattern in `ai-gateway.js` and `discord-bot/lib/ai.js` as defense-in-depth against concurrent write corruption.
