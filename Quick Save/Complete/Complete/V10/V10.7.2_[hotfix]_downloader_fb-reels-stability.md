# FB Reels Scraper Stability Hotfix

## Compiled Truth

This hotfix resolves three critical points of failure in the newly developed FB Reels Proxy Architecture (V10.7.0):

1. **Audio-Only Downloads**: Initially, the server tried to pull `bestvideo+bestaudio` and stream it via pipe. This is fundamentally flawed because DASH streaming merges require `ffmpeg`, which isn't installed on the VPS. Even with `ffmpeg`, pipe outputs (`-o -`) do not support stream merging. 
   **Fix**: Modified the format selector to `hd/sd/b[ext=mp4]/b`. These are "progressive streams" on Facebook, containing both video and audio out-of-the-box, allowing direct piping to the client with zero processing overhead.

2. **Job Queue & State Management**: Sending multiple massive Sync requests caused the node process to spin up overlapping `yt-dlp` runs, bringing the server to its knees and silently failing.
   **Fix**: Introduced a global job queue. Only one sync job runs at a time. Other jobs wait in the queue.

3. **Graceful Shutdown & Data Loss**: The sync job stored 100% of parsed data in memory, only writing to Google Sheets at the very end. If PM2 restarted the app, all progress was lost.
   **Fix**: Implemented batch flushing every 25 URLs. Also hooked into `process.on('SIGTERM')` to perform an emergency flush of whatever was pending before the app terminates.

4. **UI Improvements**: Removed the legacy `.bat` export mechanism to declutter the UI. Restyled the sidebar toggle to use a smooth CSS translation instead of abrupt `display: none;` toggles.

## Changelog

- **[Update]**: Changed Discord notification channel from `#pinky` to `#ai-monitor` (ID: 1488891766999617769) to prevent Pinky AI from incorrectly parsing system notifications as standard web-tasks.
- **[Enhancement]**: Added an 11th column `Download` to the Google Sheets integration. It generates a `=HYPERLINK(...)` formula calling the VPS `yt-dlp` endpoint directly from the sheet, allowing 1-click progressive MP4 downloads.

- **[Bug Fix]**: Fixed a fatal flaw in the Graceful Shutdown logic where `server.js` was calling `process.exit(0)` synchronously on `SIGTERM`/`SIGINT`, completely skipping the asynchronous `flushToSheets` operation in `downloader.js`. `server.js` now awaits `shutdownDownloader()` before exiting.
- **[Enhancement]**: Implemented a Persistent State Queue (`fb-reels-scraper.json`) that writes the active job progress (currentIndex) and all pending queues to disk upon shutdown. When the server restarts, it automatically reloads the queue and resumes incomplete jobs exactly from where it left off without skipping any URLs.
- **[Enhancement]**: Updated all Discord notifications to display the **Page Name** (e.g., `เพจ Masih Message`) instead of raw `job_id` strings once the page metadata is discovered, and added resume announcements.

  - Added regex parsing to extract `views` directly from the `fulltitle` string if `meta.view_count` is missing.

  - Errors (`yt-dlp error`) are no longer written to Google Sheets as garbage artifact rows. They are gracefully skipped, tracked in memory, and reported as `❌ ดึงข้อมูลล้มเหลว: X คลิป` in the final summary.

- **[Bug Fix]**: Fixed `ReferenceError: urls is not defined` crash in `processNextJob()`. During the `urls` → `items` migration, two references to the old `urls` variable were missed: the flush condition (`i === urls.length - 1` at line 374) and the inter-item delay (`i < urls.length - 1` at line 402). # FB Reels Scraper — Bug Fixes & Improvements (V10.7.2)

## Root Cause Analysis

### 🔥 Issue 1: Sync Jobs รับงานแต่ไม่เสร็จ (Critical)

จากการตรวจสอบ PM2 logs จริงบน VPS พบ **2 สาเหตุหลัก**:

#### A. Concurrent Job Stacking — งานซ้อนกัน
```
Background job started: 150 URLs → ทำได้ถึง #64/150
(ยังทำ 150 ไม่เสร็จ แต่ user กด Sync อีกครั้ง)
Background job started: 126 URLs → ทำได้ถึง #44/126
```
- User กดปุ่ม Sync หลายครั้ง → **background job หลายตัวรันพร้อมกัน**
- yt-dlp processes ซ้อนกัน → VPS ช้า/ล้มเหลว → ไม่มี job ไหนเสร็จ

#### B. Server Restart ระหว่างทำงาน
```
[sync-sheets] [76/190] Fetching: ...reel/1835601477126045/
🛑 Shutting down...  ← server ถูก restart
✅ Database schema created  ← server เริ่มใหม่ — job หายหมด!
```

#### สรุป: ทั้ง 3 batch (150, 126, 190) ล้มเหลวหมด — มีแค่ batch 1 URL และ 75 URLs ที่สำเร็จ

---

### 🔥 Issue 2: โหลด MP4 ได้แต่เสียง ไม่มีภาพ (Critical)

**Root Cause พบ 2 ปัญหาซ้อนกัน:**

#### A. ไม่มี ffmpeg บน VPS
```bash
$ which ffmpeg
ffmpeg NOT FOUND
```
→ yt-dlp ไม่สามารถ merge video + audio DASH streams ได้

#### B. Pipe mode (`-o -`) ไม่รองรับ merge อยู่แล้ว
แม้จะมี ffmpeg ก็ตาม — `-o -` (stream to stdout) **ไม่สามารถ merge 2 streams** ได้ ต้องใช้ single stream เท่านั้น

#### C. Format Selector ผิด
```js
// ❌ ปัจจุบัน — พยายาม merge (ทำไม่ได้) → fallback ได้แค่ audio
'-f', 'bestvideo[ext=mp4]+bestaudio[ext=m4a]/best[ext=mp4]/best'
```

#### Facebook Format List จริง:
```
ID                EXT RESOLUTION | VCODEC          ACODEC     MORE INFO
--------------------------------------------------------------------
867858596324619a  m4a audio only | audio only      mp4a.40.5  DASH audio
sd                mp4 unknown    | unknown         unknown    ← progressive ✅
hd                mp4 unknown    | unknown         unknown    ← progressive ✅
1462572724886318v mp4 540x960    | av01.0.04M.08   video only DASH video
2825184451155676v mp4 1080x1920  | av01.0.08M.08   video only DASH video
```

**`sd` และ `hd`** = progressive streams (video+audio รวมกันแล้ว) → ใช้ pipe ได้ ✅

## Proposed Changes

### 1. Fix MP4 Audio-Only — [downloader.js](file:///c:/My%20Claw/Openclaw-VPS/routes/downloader.js) line 274

**เปลี่ยน format selector:**

```js
// ❌ เดิม — merge mode ใช้กับ pipe ไม่ได้
spawn('yt-dlp', ['-f', 'bestvideo[ext=mp4]+bestaudio[ext=m4a]/best[ext=mp4]/best', '-o', '-', url])

// ✅ ใหม่ — เลือก progressive stream ที่มีทั้ง video+audio ในไฟล์เดียว
spawn('yt-dlp', ['-f', 'hd/sd/b[ext=mp4]/b', '-o', '-', url])
```

| Format | ความหมาย |
|--------|----------|
| `hd` | Facebook progressive HD (video+audio, ~720p+) |
| `sd` | Facebook progressive SD (video+audio, fallback) |
| `b[ext=mp4]` | Best combined single stream ที่เป็น mp4 |
| `b` | Best combined single stream ใดๆ |

**ไม่ต้องลง ffmpeg** — เพราะ progressive stream ไม่ต้อง merge

### 2. Fix Sync Job Reliability — [downloader.js](file:///c:/My%20Claw/Openclaw-VPS/routes/downloader.js)

#### A. Job Queue — ต่อคิวแทน reject

```js
let activeJob = null;
const jobQueue = [];

// เมื่อรับ job ใหม่:
if (activeJob) {
    jobQueue.push({ urls, sheetId, jobId });
    sendDiscordNotification(`📋 Job เข้าคิว (${urls.length} URLs) — รอ job ก่อนหน้าเสร็จ`);
} else {
    startJob(urls, sheetId, jobId);
}

// เมื่อ job เสร็จ → ดึง job ถัดไปจากคิวมาทำ
```

#### B. Progress Reporting — แจ้งทุก ~33% / ~66%

```js
const milestones = [Math.floor(total * 0.33), Math.floor(total * 0.66)];
if (milestones.includes(i + 1)) {
    sendDiscordNotification(`📊 Progress: ${i+1}/${total} (${pct}%)`);
}
```

#### C. Graceful Shutdown — Batch Flush ทุก 25 URLs

เปลี่ยนจาก "เก็บทั้งหมดแล้วเขียนทีเดียวตอนจบ" → **flush ลง Sheet ทุก 25 URLs**:

```js
let pendingResults = [];
const FLUSH_EVERY = 25;

// ในลูป:
pendingResults.push(result);
if (pendingResults.length >= FLUSH_EVERY || i === urls.length - 1) {
    await flushToSheets(pendingResults, sheetId, tabName);
    pendingResults = [];
}

// SIGTERM: flush ข้อมูลที่เหลือ + แจ้ง Discord
process.on('SIGTERM', async () => {
    if (pendingResults.length > 0) await flushToSheets(...);
    sendDiscordNotification(`⚠️ Server restart — บันทึกข้อมูลที่ทำเสร็จลง Sheet แล้ว`);
});
```

**ก่อน**: ตายที่ #76/190 → ข้อมูล 76 ตัวแรก**หายหมด**
**หลัง**: ตายที่ #76/190 → ข้อมูล 75 ตัวแรก**อยู่ใน Sheet แล้ว**

- Timeout: 30s → **90s**
- Retry: 1 ครั้ง ก่อนถือว่าล้มเหลว
