# Fixing Tata Context Loss and AI History Bloat (V10.3.0)

## The Problem
The user asked to review why the Tata agent was replying without knowing the context:

**User's intent:** "ช่วยรีวิว Tata agent ทำไมถามตอบ ไม่รู้บริบทเลย"

### The Incident
When the user gave a follow-up statement "ลืมบอก รอทำคืนนี้" (waiting to do this tonight) after successfully creating a task in the previous turn, Tata failed to recognize the context of the task and replied with a fallback routing rule from her prompt: `"อันนี้ไปบอกพิ้งกี้ที่ห้อง #pinky นะคะ 🎀"`.

## Root Cause Analysis
Upon investigating the AI logic in `discord-bot/lib/ai.js`, I found two intersecting issues:

1. **Context/Token Bloat (History Pollution):**
   In `lib/ai.js`, the code was fetching the `brainContext` (which is a massive dump of all Tasks, Memories, Notes, Events, Journal, and Financial summaries) and prepending it to the current user's message.
   The critical bug was here:
   ```javascript
   history.push({ role: 'user', content: fullUserMessage });
   ```
   Instead of saving only the raw `userMessage` to the history array, the bot was saving `fullUserMessage` (which includes the entire `brainContext`). Over multiple turns, the context window became overwhelmingly polluted with redundant DB states (e.g., Turn 1: DB + Message, Turn 2: DB + Message). This pushed the actual conversational context completely out of the AI's attention span.

2. **Strict Fallback Rules (`tata.md`):**
   Because the AI's attention was flooded with database dumps, it looked at the very last line `"ลืมบอก รอทำคืนนี้"`, completely disconnected it from the prior task creation, and triggered her strict routing rule for non-task chats:
   `- คุยเล่น/จดโน้ตทั่วไป ➡️ "อันนี้ไปบอกพิ้งกี้ที่ห้อง #pinky นะคะ 🎀"`

## Implementation (The Fix)
I modified `discord-bot/lib/ai.js` to correctly maintain the history:

```diff
-    history.push({ role: 'user', content: fullUserMessage });
+    history.push({ role: 'user', content: userMessage }); // ❌ STOP SAVING ENTIRE DB CONTEXT INTO HISTORY!
```
Now, `fullUserMessage` (which contains the DB dump) is *only* sent to the LLM for the *current* evaluation turn, but it is NOT permanently saved into the `history` array. The history array now remains clean and strictly conversational.
