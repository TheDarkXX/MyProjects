# ✨ Advanced AG Skills Expansion & Root Directory Cleanup

## 1. บริบทจากบทสนทนา (Context & Decision)

หลังจากการสถาปนากฎ `/skill` และไฟล์ `AG_SKILLS_INDEX.md` ในรุ่น V5.0.0 แล้ว ผู้ใช้ได้ขอให้ AG ประเมิน **พฤติกรรมที่น่าหงุดหงิด (Pain Points)** ตลอด 2-3 สัปดาห์ที่ผ่านมาที่มักจะเกิดเรื่องการเตือนและกฎ Iron Rules ซ้ำๆ AG จึงได้วิเคราะห์ประวัติและเสนอกลไกแก้ไขเป็นซีรีส์สกิลรวดเดียวจบ

นี่คือไทม์ไลน์ของการตัดสินใจ:
1. **การรวมไฟล์ `/save`:** ตอนแรก AG แยกส่วนประกอบเป็น `/deploy`, `/quicksave`, `/markdone` ทว่าผู้ใช้อนุมัติให้ **รวบเป็น `/save` ท่อเดียว** ที่จัดการตั้งแต่กวาด Context, เซฟหน้ามาร์กดาวน์, เตือนชน VS Code, ทะลวงบัมพ์แคช, และดันขึ้น VPS สู่ Production
2. **ปรับกลยุทธ์ Recheck & Testing:** เพื่อป้องกันอีโมจิแตก และลืมทดสอบฟังก์ชัน User ระบุให้แยกเทสต์เบราว์เซอร์ออกไปให้ชัดเจน
   - เลยได้มาเป็น `/recheck` (ตรวจโค้ดเบื้องต้น แก้อีโมจิ)
   - `/webtest` (สปินเบราว์เซอร์ไปทดสอบหน้า Web Production)
   - `/posttest` (E2E Functional test ส่งข้อมูลจำลองทะลวงบอท)
3. **การทำความสะอาดโฟลเดอร์ Root (`/junk`):** Root Directory มีไฟล์ `scratch_*.js` และ `tmp_*.js` เกลื่อนกลาด จึงสร้างสคริปต์ `/junk` แบบบังคับ Double Check ก่อนลบ 
4. **กระบวนการ `/fail` (แทนการพิมพ์ JSON เอง):** สร้างเพื่อแก้ป้ัญหาตารางเรียนรู้ความผิดพลาด (skill-fail) พิมพ์ยาก ยุ่งยาก 

และระหว่างทาง AG พบว่าการอัปเดต Quick Save ทะลุเลข V5.0.0 เมื่อครู่ทำให้ครบเงื่อนไข **Rolling Archive Rule** พอดี จึงสร้างกล่องย่อย `/V4/` และกวาดประวัติ V4 เข้าไปทั้งหมดทันที

## 2. โครงสร้างที่นำไปใช้งาน (Implemented Changes)

### 2.1 The 6 New Omnipotent Skills
เพิ่มไฟล์เข้าไปใน `docs/skills/` ควบคู่กับการประกาศลง Target File ใน `AG_SKILLS_INDEX.md`:
* `save.md`: The Ultimate end-of-task pipeline.
* `recheck.md`: Post-Implementation Code checking tools.
* `webtest.md`: Production browser DOM subagent tester.
* `posttest.md`: E2E Integration and payload transmission tester.
* `junk.md`: Scans `tmp_*` and `scratch_*` then cleanly `Remove-Item` safely.
* `fail.md`: Dynamically formats unstructured user failure complaints into strict `skill-emit.jsonl` formats.

### 2.2 Root Reorganization (โฟลเดอร์มีระเบียบ)
มีไฟล์สคริปต์และ SQL อยู่เต็มหน้าแรก จึงทำการย้ายเข้าตามหลักดังนี้:
- `*.sql` -> `scripts/sql/`
- `*.py` -> `scripts/python/`
- `ocl-help.txt` -> `docs/`
- `SOUL*.md` -> `agents/core-souls/`

## 3. ทำไมถึงเลือกแนวทางนี้แบบนี้ (Architectural Insight)

- **One-Button Paradigm:** เราลดภาระของ User Prompting ทิ้งทั้งหมด ความเสี่ยงที่ผู้ใช้หรือ AG จะหลงลืมสเต็ปอย่างเช่น `node bump-cache.js` หมดไปพร้อมกับคำสั่ง `/save`
- **Double Recheck vs Automated:** สิ่งใดที่เป็นอันตรายต่อสภาพแวดล้อมโดยตรง (เช่น `/junk`) ได้ถูกออกแบบหน้าด่านให้มี "MANDATORY PAUSE" ชัดเจน ในขณะที่กฎอื่น ๆ ที่เน้น Output Production ปล่อยไหลรวดเดียวจบได้เลย
- **Rolling Archive Enforcement:** การสอดแทรกคำเตือนการปัดเลข V ลงในขั้นตอนที่ 4 ของการ `/save` ทำให้ AG จะไม่มีการลืมสร้างโฟลเดอร์เก็บเข้ากรุยุคเก่าอีกเลย
