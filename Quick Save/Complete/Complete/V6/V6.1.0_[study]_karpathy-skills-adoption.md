## Source
- **Repo:** [forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
- **Stats:** ⭐ 53.8k stars, 4.5k forks
- **What it is:** A single `CLAUDE.md` file with 4 behavioral principles to reduce common LLM coding mistakes
- **Based on:** [Andrej Karpathy's X post](https://x.com/karpathy/status/2015883857489522876) criticizing LLM coding behavior

## Karpathy's Original Observations (Verbatim)

> "The models make wrong assumptions on your behalf and just run along with them without checking. They don't manage their confusion, don't seek clarifications, don't surface inconsistencies, don't present tradeoffs, don't push back when they should."

> "They really like to overcomplicate code and APIs, bloat abstractions, don't clean up dead code... implement a bloated construction over 1000 lines when 100 would do."

> "They still sometimes change/remove comments and code they don't sufficiently understand as side effects, even if orthogonal to the task."

## The 4 Karpathy Principles — Full Detail

### Principle 1: Think Before Coding
**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### Principle 2: Simplicity First
**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### Principle 3: Surgical Changes
**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

### Principle 4: Goal-Driven Execution
**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

## Anti-Patterns Examples (from EXAMPLES.md)

### Hidden Assumptions Anti-Pattern
**User says:** "Add a feature to export user data"
**LLM does wrong:** Assumes ALL users, assumes file location, assumes JSON format, assumes which fields
**Correct:** Ask about scope, format, fields, and volume BEFORE implementing

### Over-abstraction Anti-Pattern
**User says:** "Add a function to calculate discount"
**LLM does wrong:** Creates Strategy pattern with ABC, Enum, DiscountConfig, DiscountCalculator (30+ lines setup)
**Correct:** Single function: `calculate_discount(amount, percent)` — 3 lines

### Drive-by Refactoring Anti-Pattern
**User says:** "Fix the bug where empty emails crash the validator"
**LLM does wrong:** Also changes comments, adds docstrings, adds username validation, improves email regex
**Correct:** Only change the 2-3 lines that fix the empty email handling

### Style Drift Anti-Pattern
**User says:** "Add logging to the upload function"
**LLM does wrong:** Changes quote style, adds type hints, reformats whitespace, changes boolean logic
**Correct:** Only add logger.info/error lines, matching existing single-quote style

### Vague vs. Verifiable Anti-Pattern
**User says:** "Fix the authentication system"
**LLM does wrong:** "I'll review the code, identify issues, make improvements, test changes" (no criteria)
**Correct:** "What specific issue? If 'users stay logged in after password change': 1. Write test → verify fails 2. Fix → verify passes 3. Check edge cases → verify passes"

**Summary Table:**

| Principle | Anti-Pattern | Fix |
|-----------|-------------|-----|
| Think Before Coding | Silently assumes file format, fields, scope | List assumptions, ask for clarification |
| Simplicity First | Strategy pattern for single discount calculation | One function until complexity needed |
| Surgical Changes | Reformats quotes, adds type hints while fixing bug | Only change lines that fix the issue |
| Goal-Driven | "I'll review and improve the code" | "Write test → make pass → verify" |

### Principle 1: Think Before Coding — PARTIAL ADOPT

**Conflict:** Our Core Rule #1 "Do the work — never just talk about it" is the OPPOSITE philosophy. Karpathy = accuracy-first. Our system = speed-first.

**Real-world problems this would fix:**
- Hydra Evolver 0% success rate — assumed what to change without validating
- memory.md corrections: wrong URLs, wrong passwords, wrong paths

**Decision:** Adopt as CONDITIONAL rule only:
- ✅ APPLIES WHEN: 5+ files changed, first-time touching subsystem, Hydra auto-proposal
- ❌ DOES NOT APPLY: Trivial tasks, hotfixes, single-file edits

### Principle 2: Simplicity First — FULL ADOPT (P0)

**Why highest priority:** No anti-overengineering rule existed anywhere in our system. We had:
- `creditcard-triple-guard-consensus` — 3 AI models validating each other
- Hydra: 4 meta-agents when 2 would suffice
- Notification system: 3 iterations (real-time → buffer → digest) from over-engineering day 1

**Nuance:** AI validation multi-layer IS intentional (hallucination is real). Simplicity applies to infrastructure/meta-agent code, not to data integrity guards.

### Principle 3: Surgical Changes — FULL ADOPT (P0)

**Why perfect fit:** Low conflict with existing rules. Directly addresses:
- Evolver proposals failing from changing too much
- Hotfixes caused by adjacent code changes
- Set-Content encoding incident (used different tool than existing pattern)
- CSS style drift in brain-app-public

### Principle 4: Goal-Driven Execution — ENHANCE EXISTING (P2)

**Already 60% done:** Planning Mode Phase 5, Post-Implementation Verification, task.md checkboxes, Browser Test protocol, SSH verify after git push vps.

**Enhancement:** Add per-step `→ verify:` pattern to task.md format.

## ⛔ Karpathy Principles — Iron Rules (NEVER violate!)

After every implementation, before reporting completion:
1. Count lines changed. If > 100 lines for a single feature → ask: "Could this be 50?"
2. No abstractions for single-use code
3. No features, config, or error-handling beyond what was explicitly asked

### Surgical Traceability (ALWAYS applies)
Every changed line must trace directly to the user's request:
1. Don't "improve" adjacent code, comments, or formatting
2. Don't refactor things that aren't broken
3. Match existing code style, even if you'd do it differently
4. If you notice unrelated dead code → **MENTION** it, don't delete it
5. Remove only what YOUR changes made unused

**APPLIES WHEN:** 5+ files changed, first-time touching a subsystem, or Hydra auto-proposal

Before implementing:
1. State your assumptions explicitly (1-2 sentences max)
2. If multiple interpretations exist, list them — don't pick silently

**DOES NOT APPLY:** Trivial/known-path tasks, hotfixes, single-file edits

## Key Insight from Karpathy

> "LLMs are exceptionally good at looping until they meet specific goals... Don't tell it what to do, give it success criteria and watch it go."

This is the core philosophy: **transform imperative instructions into declarative goals with verification loops.**

**How to know the guidelines are working:**
- Fewer unnecessary changes in diffs — Only requested changes appear
- Fewer rewrites due to overcomplication — Code is simple the first time
- Clarifying questions come before implementation — Not after mistakes
- Clean, minimal PRs — No drive-by refactoring or "improvements"

**Good code is code that solves today's problem simply, not tomorrow's problem prematurely.**
