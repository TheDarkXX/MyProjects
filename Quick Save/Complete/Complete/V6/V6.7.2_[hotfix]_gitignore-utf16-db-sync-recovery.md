# Hotfix: Gitignore Encoding Corruption & Secondary Data Recovery

## 1. Issue Description

The Ghost Bug struck again today: The user reported "วันนี้ข้อมูล db หายอีกแล้ว" (Today the db data is lost again) immediately after the Dev Agent pushed changes to the repository ("Synchronizing Local Repository State" at ~12:51). The exact same tasks restored yesterday (แว่นตาว่ายน้ำ Tiger and ค่าหนังสือ) vanished once more.

## 2. Root Cause Analysis

- **The Gitignore Encoding Failure:** During yesterday's hotfix (V6.7.1), the strings appended to `.gitignore` to block `brain.db` were silently corrupted with spaces (`t m p / a i - s t a t u s . j s o n`, `d b / b r a i n . d b`). This happens when text is saved/appended with UTF-16 LE encoding natively by certain Windows tools, injecting null bytes between characters.
- **The Tracking Resurgence:** Because ` d b / b r a i n . d b ` is not a valid gitignore pattern for `db/brain.db`, the database remained tracked in the local Git repository!
- **Data Deletion Vol. 2:** When the user/agent executed a `git push vps master` today near 12:51, the deploy hook `git checkout -f` on the VPS forcibly overwrote the live `brain.db` with the tracked version from the local machine. The local machine's DB did not contain yesterday's manual SQL injections (tasks 49, 50), so they were erased from production again.

## 3. Implementation and Final Fix

1. **Cleaning `.gitignore` locally:**
   - Modified `.gitignore` to replace all spaced-out characters with clean ASCII.
   - Executed `git rm --cached ...` on `db/` files again locally.
   - Committed the fix.

2. **Crash-Safe Deployment on VPS:**
   - Realizing that `git push vps` would now push a tree *deleting* `brain.db`, I ran a preemptive backup of `/root/brain-app/db` on the VPS to `/root/db_backup_after_crash`.
   - Executed `git push vps master` which triggered the deploy hook.

3. **Data Merging and Restoration (On VPS):**
   - The backup from the pre-push window safely contained today's new data (Tasks 51, 52).
   - Moved the backup DB back into production via `cp -p /root/db_backup_after_crash/* /root/brain-app/db/`.
   - Re-executed yesterday's `restore_tasks.sql` script directly on the restored VPS database (`sqlite3 brain.db < restore_tasks.sql`).
   - The lost tasks were successfully merged into the live database. Tasks previously known as 49 and 50 auto-incremented into slots **53** and **54**, perfectly recovering the missing data without damaging today's new items.

## 4. Retrospective / Iron Learnings

Always verify `.gitignore` via `git ls-files` directly after attempting to hide a file. If `git ls-files` still outputs the file path, the ignore block is either out of scope or corrupted by encoding. The VPS production state is now truly immune to git hooks.
