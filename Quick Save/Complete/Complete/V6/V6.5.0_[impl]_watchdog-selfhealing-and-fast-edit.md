# Watchdog Self-Healing & Discord Fast-Edit Fast-Patch

## 1. Background & Objectives
The system suffered from `cron-auto-planner.js` silently failing due to missing modules (`off-peak-agents`). To resolve this and establish long-term robustness, the user mandated a "Zero Manual Involvement" strategy by introducing autonomous self-healing capabilities.
Following that success, the user requested the ability to order simple hotfixes directly via Discord, inspired by the primary `Openclaw` operations model limitating the need to switch to the PC terminal.

## 2. Core Decisions & Execution

### Phase A: Watchdog "Doctor AI" Self-Healing
1. **Trigger Condition:** Watchdog now counts failures. If a cron script fails 3 times, it trips the `escalation` flag.
2. **Remediation Action:** `ai-remediate-planner` was built into `watchdog/remediate.js`.
3. **Model Selection:**
   * Initially utilized `Gemini Flash`.
   * **Decision:** Shifted to Evo-style `MiniMax M2.5` (`callMiniMax25` added to `api-clients.js`) as the primary Node.js surgeon for accuracy.
   * **Fallback:** If `MiniMax M2.5` fails, it falls back to `Gemini Flash`.
4. **Post-Check Bug Fix:** Resolved a `SyntaxError: Identifier has already been declared` caused by un-scoped `const` inside `switch/case` statements by wrapping case logic in `{ ... }` blocks. Added `timeout: 120000` to `execSync` to prevent frozen event loops.

### Phase B: Discord `!edit` (Fast-Edit Agent)
The goal was to enable users to type `!edit [prompt]` in `#dev-agent` Discord and have the codebase patch itself without opening VS Code.

**Architecture:**
- Discord Bot captures `!edit <prompt>` and spawns a child process: `node scripts/hydra/discord-patch-agent.js`.
- The `discord-patch-agent` uses **Gemini 2.5 Flash** to scan the directory tree and predict which 1-2 files to target.
- The agent reads those files and feeds them to **MiniMax M2.5** to draft a surgical string replacement JSON proposal.

**Risk Management (Safety Layer):**
A critical design choice was adding a `risk_level` parameter.
- **Low Risk / Safe:** (Typo fixes, color changes, basic UI) -> Agent immediately triggers `evolver.js` to apply the patch and deploy.
- **Medium / High Risk:** (Logic, DB routes, deleting functions) -> Agent halts deployment and saves an `approved` proposal in DB. It logs `STAGED_FOR_APPROVAL`.
- Discord Bot parses `STAGED_FOR_APPROVAL` and warns the user: 
  > ⚠️ คำสั่งมีความเสี่ยงระดับ HIGH! ระบบดักรอไว้เพื่อความปลอดภัย. พิมพ์ `!evo approve [ID]` เพื่อยืนยัน

#### Implementation Details (Bug Fixes applied during dev):
1. **Top-Level Crash Fix:** `db` initialization in `discord-patch-agent.js` was moved inside `run()` to prevent silent, early unhandled errors if DB paths failed.
2. **Evolver Timeout:** Added `timeout: 90000` to the `evolver.js` execution to keep Discord responsive.
3. **Help Update:** Updated `!help` in `discord-bot/index.js` to include the `!edit` command documentation.

## 3. Karpathy Sweep Compliance
- Changes were tightly scoped to the request.
- Simplicity was maintained by reusing `evolver.js` for applying code instead of building a secondary patcher inside the discord bot.
- All code additions trace directly to self-healing scaling requirements.

## 4. Next Steps & Known Limitations
- The `fast-edit` agent currently assumes it can accurately target the correct file via Gemini; if it grabs the wrong file, the `find` string will fail to match, causing no changes.
- The system correctly isolates `ideator` logic into a single micro-agent `discord-patch-agent` ensuring the core `evolver.js` logic did not bloat.

**Status:** Tested syntax, checked scopes, and deployed to VPS ✅
