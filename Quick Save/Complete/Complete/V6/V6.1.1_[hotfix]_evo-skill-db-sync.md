# 🩹 V6.1.1 — Fix /evo Skill Syntax & Hydra DB Sync

## 🎯 Context
The user observed that after using the `/evo` skill, the status of the proposals on the Hydra Awakened Protocol Web UI was not automatically updating. The items remained stuck in "Needs Review" even though they had been successfully processed by the agent.

## 🔍 Investigation & Root Cause
1. **The PowerShell Quoting Bug:** The `/evo` skill instructions in `docs/skills/evo.md` mandated the agent to run an SSH command to update the VPS SQLite database:
   ```bash
   ssh root@185.250.38.247 "sqlite3 /root/brain-app/db/brain.db \"UPDATE evolution_log SET status='approved' WHERE proposal_id='EVO-XXX';\""
   ```
   However, Windows PowerShell incorrectly parses nested escaped double-quotes (`\"`) inside strings passed to external executables, resulting in an `unexpected EOF` bash error on the VPS. 
2. **Silent Failure:** Because the `evo` workflow proceeds despite the SSH command failing (it does not enforce a halt on error), the agent thought it had completed the sync but the VPS database was never touched, leaving the proposals marked as `awaiting_human`.

## 🛠️ Implementation

### 1. Manual State Correction
Created and executed a manual SQL patch (`update.sqls`) on the VPS to resolve the stuck proposals:
```sql
UPDATE evolution_log SET status='deployed' WHERE status IN ('proposed', 'awaiting_human');
```
This correctly cleared the 5 hanging records, bringing the Web UI's "Needs Review" column down to `0`.

### 2. Preventative Instruction Update
Modifed `docs/skills/evo.md` to explicitly forbid nested quotes and instead enforce a safe, multi-step execution pattern via temporary file transfer:
```markdown
- **CRITICAL POWERSHELL RULE:** Nested quotes in `run_command` across SSH always break on Windows. You MUST:
  1. Use `write_to_file` to create a local `c:\My Claw\Openclaw-VPS\scratch\update_evo.sql` containing: `UPDATE evolution_log SET status='approved' WHERE proposal_id='EVO-XXX';`
  2. Upload it: `scp "c:\My Claw\Openclaw-VPS\scratch\update_evo.sql" root@185.250.38.247:/root/update_evo.sql`
  3. Apply it: `ssh root@185.250.38.247 "sqlite3 /root/brain-app/db/brain.db < /root/update_evo.sql"`
```
This guarantees that future `/evo` runs will correctly sync the database regardless of PowerShell's quoting quirks.
