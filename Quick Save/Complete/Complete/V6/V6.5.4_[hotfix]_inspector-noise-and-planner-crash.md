## Context
The user reported that the Hydra meta-agent's \`Inspector Report\` kept repeatedly spamming Discord about the exact same errors from \`watchdog.log\` (specifically \`cron-tracker-check\`, \`ideas-in-queue\`, \`today-has-scheduled\`), even if those errors were no longer actively occurring. They also observed that \`ideator\` was receiving budget exceeded alerts and the \`cron-auto-planner.js\` script was potentially crashing.

## Investigative Findings
1. **Inspector Notification Loop**:
   - In \`scripts/hydra/inspector.js\`, the \`Recurring Error Pattern\` check grouped Watchdog FAIL logs from the last 3 days and flagged any check_id with 3 or more errors.
   - **The Bug**: It did not verify whether the MOST RECENT run for that \`check_id\` was successful. Even if an error was fixed, it would still appear in the 3-day window and trigger a "requires fixing" report indefinitely until the 3 days elapsed.
2. **Auto-Planner Crash**:
   - An automated Evo proposal (\`EVO-20260417-166\`) uncommented imports for \`runOffPeakContentGenerator\` and \`runOffPeakMaintenance\` in \`cron-auto-planner.js\` but the files/paths did not exist locally or on the VPS. 
   - This caused the script to hard-crash on execution, halting the backlog queue pipeline entirely resulting in the \`ideas-in-queue\` drop to 0, which correctly triggered the Watchdog warning.

## Implementation Steps
### 1. Fixed \`inspector.js\` (Noise Reduction)
Updated the SQL CTE logic to ensure a recurring error is ONLY flagged if the **latest status** for that specific \`check_id\` is still a \`FAIL\`.
```sql
      WITH RankedLogs AS (
        SELECT check_id, status, message,
               ROW_NUMBER() OVER (PARTITION BY check_id ORDER BY timestamp DESC) as rn
        FROM watchdog_logs
        WHERE timestamp > datetime('now','-3 days')
      ),
      LatestStatus AS (
        SELECT check_id, status as latest_status FROM RankedLogs WHERE rn = 1
      )
      SELECT w.check_id, w.message, COUNT(*) as cnt
      FROM watchdog_logs w
      JOIN LatestStatus ls ON w.check_id = ls.check_id
      WHERE w.status = 'FAIL' 
        AND ls.latest_status = 'FAIL'
        AND w.timestamp > datetime('now','-3 days')
      GROUP BY w.check_id, w.message HAVING cnt >= 3
```

### 2. Fixed \`cron-auto-planner.js\` (Crash Prevention)
Commented out the incomplete/missing imports and function calls related to \`off-peak-agents\` to restore the pipeline functionality.
```javascript
// import { runOffPeakContentGenerator } from './off-peak-agents/off-peak-content-generator.js';
// import { runOffPeakMaintenance } from './off-peak-agents/off-peak-maintenance.js';

// ...
// await runOffPeakContentGenerator();
// ...
// await runOffPeakMaintenance();
```

## Validation
- \`watchdog\` logs show current status correctly, and \`cron-auto-planner\` will no longer instantly crash upon initialization.
- \`inspector\` will only alert if the failure is ongoing today and not merely historical artifact noise.
