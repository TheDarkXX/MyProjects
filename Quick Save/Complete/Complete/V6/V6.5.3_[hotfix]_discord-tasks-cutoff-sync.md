# Hotfix: Discord Task "Silent Fail" Persistence

## 1. Context & Rationale
The user reported a recurring issue over the past few days where Discord bots (TaTa, Dev Agent) would acknowledge a task (`✓ Task added: ...`) but the task would seemingly never enter the system. The tasks did not show up in the Web UI, nor did they appear in the Daily/Morning AI summary.

## 2. Root Cause Analysis
1. **API Hit Success:** The bots correctly parsed user intents, executed `POST /api/tasks`, and received a HTTP `201 OK`. We proved the tasks *were* indeed being inserted cleanly into the production SQLite `brain.db` database.
2. **The Cutoff Bug:** 
   - The `GET /api/tasks` endpoint had a hardcoded default `LIMIT 50`.
   - The query did not exclude `status = 'done'` or `status = 'cancelled'` unless explicitly requested via `due=overdue`.
   - Because `ORDER BY ... due_date ASC` treats `NULL` due dates as numerically smaller, tasks without due dates correctly floated to the top. However, since older `P1` tasks that were already `done` accumulated over time and didn't get filtered out, they permanently ate up the 50 slots.
   - Consequently, when the AI agent fetched `brainQuery(['tasks'])` for its Morning Brief, it only received a clipped array of 50 ancient, completed tasks, leaving out any freshly inserted tasks over the limit.

## 3. Implementation (The Fix)
The solution was to completely remove completed tasks from the default "Active" quota pool, rather than just pushing them down, effectively guaranteeing UI and AI contexts are strictly filled with actionable data.

1. **API Default Exclusion (`routes/tasks.js`):** 
   - `GET /api/tasks` now natively injects `AND status NOT IN ('done', 'cancelled')` unless a specific status is explicitly requested, or `include_done=1` is provided.
   - Buffer limit increased from `50` to `500` to prevent any theoretical starvation of active tasks.
   
2. **Web UI Synchronisation (`brain-app-public/tasks.js`):**
   - Modified the `loadTasksData` client fetch URL. If no explicit status filter is set (default view), it appends `&include_done=1`.
   - This allows the `done-section` expansion panel to continue working exactly as before while ensuring active tasks aren't mixed into the same quota.
   
3. **Daily Review Synchronization (`discord-bot`):**
   - Morning Brief / Follow up: Relies on the new default API behavior to pull *only* Active tasks (`todo`/`doing`).
   - `generateDailyReview()`: Modified the task-fetch wrapper (`lib/morning-brief.js`) to pass a third boolean flag to `brainQuery(['tasks', null, true])`.
   - `getTasks` (`lib/brain.js`): Appends `?include_done=1` when this flag is activated, ensuring the `✅ เสร็จวันนี้` section in the Evening Digest actually receives the tasks you checked off today.

## 4. Stability Check
- **No data loss:** No new parsing logic was introduced, the `POST` route was entirely stable. The issue was purely read-time visibility logic.
- **Web UI & Discord Restored:** The Task Dashboard (which uses `limit=20`) now safely renders the *actual* Top 20 active tasks. The Discord Agents fetching `brainQuery(['tasks'])` will again "see" new tasks properly and include them in the Daily/Follow-up LLM briefs.
