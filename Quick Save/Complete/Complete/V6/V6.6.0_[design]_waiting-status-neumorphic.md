# Feature & Design Sync: Waiting Status Lifecycle

## 1. Architectural Reasoning
A core problem in task velocity is mixing "Actionable" tasks with "Blocked" ones. When waiting on a 3rd party (e.g., waiting for merchant replies), pushing the task to `doing` clutters the workspace, but leaving it in `todo` hides the context. 
We introduced a native `waiting` status.

## 2. Strict UI Design System Application
Following `MyDesign/Design System Brain APP Dark Dashboard.md`, we aligned the new status with the 'Pending / Warning' color family:
- **Burnt Orange:** `--o5` (`#FD5514`) as the anchor dot.
- **Support Brightness:** `--o4` mapping applied to textual components (`#FF7A30`) with a highly transparent dark-mode compliant active background (`rgba(253,85,20,0.12)`).

## 3. The Backend Expansion
- Validated logic in `routes/tasks.js` to ensure that `waiting` isn't accidentally pruned from API responses like `done` or `cancelled`.
- Registered `waiting` safely into `GET /api/tasks/stats` ensuring accurate metric computation without causing database crashes.
- Extended the `ag-grid` cell editor enumerations directly permitting frontend toggles.

## 4. NLP / Discord Integration (AI Agent Layer)
- **Tata's Prompt Engine:** Injected explicit training into `discord-bot/prompts/tata.md` coaching the AI on when to dispatch `waiting` statuses via `brain-query.sh update-task` and forcibly appending context (the "why" we are waiting) into the `description` payload.
- **Reporting Triage:** Upgraded both the Morning Brief and Evening Digest payloads (`lib/morning-brief.js`). They now feature an isolated section: `⏳ รอติดตามผล (Waiting)` designed explicitly to *exclude* tasks from the actionable Todo buckets, significantly reducing visual cognitive load while keeping tracking alive.
