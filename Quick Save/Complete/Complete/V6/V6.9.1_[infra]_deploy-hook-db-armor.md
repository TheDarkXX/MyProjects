# Infrastructure Fix: Deploy Hook Database Armor

> **"Assume the environment is hostile and the repository will try to wipe the DB again."**

## 1. Issue Recap

The Ghost Bug of lost tasks directly correlated to `git checkout -f` hooks forcibly merging local DB snapshots with the live server. During our investigation, it was discovered that:
1. `db/brain.db` was still being tracked due to encoding misconfigurations (`UTF-16` null bytes acting as spaces).
2. Even after resolving the encoding, stray artifacts like `db/brain.zip` remained tracked.
3. The VPS bare repository `post-receive` script had zero failsafes for critical files if `.gitignore` failed. 

## 2. Refinements Made Locally

- **Gitignore Hygiene:**
  Wiped out duplicate sections created during hotfixes and re-established explicit wildcards protecting all DB-related files (`brain.db`, `-wal`, `-shm`, `.zip`, `.sql`, etc.).
- **Untrack Sweep:**
  Explicitly issued `git rm --cached` on the 3 redundant DB files still attached to the tree. 

## 3. The VPS Deploy Hook (The Iron Guard)

The most pivotal upgrade is replacing the brittle `post-receive` checkout command with a 3-stage armored routine inside the VPS hook:

```bash
#!/bin/bash

# === 1. CRITICAL: Backup DB BEFORE checkout ===
DB_DIR=/root/brain-app/db
DB_BACKUP=/root/brain-app/db/brain.db.pre-deploy
if [ -f "$DB_DIR/brain.db" ]; then
  cp -p "$DB_DIR/brain.db" "$DB_BACKUP"
  cp -p "$DB_DIR/brain.db-wal" "$DB_DIR/brain.db-wal.pre-deploy" 2>/dev/null
  cp -p "$DB_DIR/brain.db-shm" "$DB_DIR/brain.db-shm.pre-deploy" 2>/dev/null
fi

# === 2. Checkout code (Danger Zone) ===
GIT_WORK_TREE=/root/brain-app git checkout -f

# === 3. CRITICAL: Restore DB AFTER checkout ===
if [ -f "$DB_BACKUP" ]; then
  cp -p "$DB_BACKUP" "$DB_DIR/brain.db"
  cp -p "$DB_DIR/brain.db-wal.pre-deploy" "$DB_DIR/brain.db-wal" 2>/dev/null
  cp -p "$DB_DIR/brain.db-shm.pre-deploy" "$DB_DIR/brain.db-shm" 2>/dev/null
  echo '[DEPLOY HOOK] DB protected — restored from pre-deploy backup'
fi
```

### Result:
Even if a developer maliciously or mistakenly forces `git add -f db/brain.db` and pushes it, the VPS will save its own production database in flight before the git command can overwrite it, and then instantly restore the production data over the pushed copy.

**Production data is now permanently decoupled from repository sync side-effects.**
