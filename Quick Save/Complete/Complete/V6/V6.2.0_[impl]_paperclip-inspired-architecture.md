# Implementation Plan: Hydra V6.0 (Paperclip-inspired)

## 🎯 Context
สืบเนื่องจากผลการศึกษา Paperclip (V5.4.0) เราพบว่าระบบ Approval Gates ดั้งเดิมของเรา (Git Piggyback + `awaiting_human`) นั้นดีและสอดคล้องกับโครงสร้างอยู่แล้ว สิ่งที่เราต้องนำมา Implement ใน V6.0 เพื่ออุดช่องโหว่การเป็น "Corporate Control Plane" จริงๆ คือ 2 ส่วน:
1. **Budget Enforcement Guard** (`lib/budget-guard.js`) — ระบบเบรกเกอร์อัตโนมัติ ตัดการทำงานถ้า Agent ถลุง Token รายวัน/รายเดือน เกินโควตาที่จำกัดไว้ (ป้องกัน Runaway Costs)
2. **Heartbeat Context Injection** (`lib/heartbeat.js`) — ห่อหุ้มก่อนที่ Agent จะตื่นมารับงาน ให้ไปดึง Context/สถานะล่าสุดมาฉีดให้ก่อน เพื่อแก้ปัญหา Agent ความจำเสื่อม (Statelessness)

## 🚀 Execution Logs & Architectural Decisions

**บริบทการตัดสินใจจากบทสนทนา (Conversation Context):**
1. **Approval Gates** (Paperclip concept) เราค้นพบกลางทางว่า V5.3.x มีอยู่แล้วในไฟล์ `evolver.js` (การจับสถานะ `awaiting_human` แล้ว dump file ลง `_incoming/` พร้อมบล็อค `auto-deploy`) ทำให้ไม่ต้องไปอิมพลีเมนต์โครงสร้าง approval gates ซ้ำซ้อนใหม่ และตรงกับรูปแบบ Git Piggyback ของเราได้เนียนกว่า 
2. **Budget Rule:** จากเดิมตอนแรกตั้งใจทำ budget_guard อิงกับ `kie-proxy/logUsage` แบบ DB แต่เปลี่ยนมาใช้วิธี **อ่าน `discord-bot/data/usage.json` ทื่อๆ** แทน เนื่องจากประหยัด overhead ค้นหาใน proxy proxy ทำให้เช็ค Token แบบ real time ได้รวดเร็ว โดยคำนวณราคาเองจากตาราง Hardcode ($1.20 สำหรับ m2.7, $0.90 สำหรับ gemini, etc.)
3. **Breaker Cutoff:** `ideator.js` เสี่ยงจะปั๊ม token ฟรีๆ หากมันรันตอนที่ Evolver อนุมัติไม่ทันเกิน 5 งาน (Proposals Waiting >= 5). เราจึงตั้ง Breaker ดับเครื่อง Ideator อัตโนมัติ (`skipped_queue_full`) และส่ง Discord Alert ถ้าอันไหนทะลุ limits 
4. **Heartbeat Fix:** ตอน /recheck เราเจอบั๊ก `ideator.js` ที่ return ทิ้งทันทีหาก budget พังโดยไม่ได้ `db.close()` จนมันกลายเป็น DB connection leak ตรงนี้ได้รับการแพทช์ในบิลด์สุดท้ายเรียบร้อย 

**บทสรุป:** Hydra V6.0 ถือเป็นการยกเครื่อง Control Plane เพื่อจำกัดต้นทุนให้เป็นกิจจะลักษณะได้สำเร็จ และอิงแนวทาง Paperclip ในวิธีที่ Simple ที่สุดตามฉบับ Karpathy.
