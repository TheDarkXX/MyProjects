# 🧠 V5.3.5 — `/evo` Skill Web UI Database Sync & Archiving

## 🎯 Context
During the completion of **V5.3.4 (Evolver Validation)**, the user inquired about the fate of the `_incoming/hydra/` proposal stubs after running the `/evo` skill, and correctly pointed out a systemic gap: The `/evo` skill was managing file states locally, but not informing the VPS Web UI of the final decision. This left proposals visibly stuck in the Dashboard as "Awaiting Human".

## 🛠️ The Implementation Plan

1. **Modify Skill Manual (`docs/skills/evo.md`)**
   - **Step 6.4 (Archiving):** Instruct AG to stop deleting the `.md` stubs from `_incoming/hydra/`. Instead, route them to `Archive/Approve/` or `Archive/Reject/` to preserve proposal history locally (Audit trail). Created the directories explicitly.
   - **Step 7 (Database Sync):** Instructed AG to auto-sync the decision state to the server upon approval/rejection. Added explicit SSH SQLite injection sequence: `ssh root@185.250.38.247 "sqlite3 /root/brain-app/brain.db \"UPDATE evolution_log SET status='approved' WHERE proposal_id='<ID>';\""`.

2. **Backend Validation:**
   - Pre-verified the existence of Web UI POST endpoints (`routes/hydra.js`: `/proposals/:id/approve`). 
   - Concluded SSH database manipulation is superior for the `dev-agent` since AG operates safely alongside production CLI.
