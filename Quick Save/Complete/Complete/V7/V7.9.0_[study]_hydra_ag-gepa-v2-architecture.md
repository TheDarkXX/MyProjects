# AG-GEPA V2: Concept Adoption Roadmap

## Context & Problem
We recently shipped AG-GEPA (V7.8.0) as an internal evolutionary engine inspired by Hermes. However, new major industry repositories (`imbue-ai/darwinian_evolver` and `gepa-ai/gepa` backed by major teams) provide robust capabilities that our initial implementation lacks. Rather than rip out our lightweight Node.js engine, we need an architectural roadmap to upgrade our engine (AG-GEPA V2) with their best concepts.

## Implementation Strategy / Rationale

We performed a deep analysis and the user approved adopting concepts from both while keeping our cost-efficient, natively integrated, risk-gated engine.

**From `gepa-ai/gepa` (Approved concept adoption):**
1. **Actionable Side Information (ASI):** Move beyond binary pass/fail and standard numeric scores. Have the evaluator output full execution logs, errors, and profiling data as diagnostic text.
2. **System-Aware Merge:** Rather than picking a single winner, implement crossover. Track the Pareto frontier and allow the LLM to merge two variations that excel in different cases.
3. **Reflective Dataset Construction:** Log comprehensive execution histories so the Reflector LLM can evaluate an entire timeline of tests to create a more robust diagnosis.

**From `darwinian_evolver` (AGPL Iceboxed, concepts adopted):**
1. **Failure Case Routing:** Pinpoint exact test cases that failed and feed ONLY those into the Mutator to decrease token cost and sharpen focus.
2. **Post-Mutation Verification:** Implement a fast, cheap preliminary check before executing a full expensive evaluation benchmark. 
3. **Novelty Bonus:** Add a score bump to parents that have seldom been mutated, escaping local optima.
