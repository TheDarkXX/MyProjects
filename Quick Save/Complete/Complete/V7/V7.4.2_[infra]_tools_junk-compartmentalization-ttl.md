# 🧹 V7.4.2: Strict Compartmentalization & Smart Junk TTL

## 📌 Context & Problem
During the execution of the `/junk` command, over 20 temporary development scripts (`test_api.js`, `scratch-query.js`, `close-jobs-final.js`, etc.) were discovered mixed into the project's root folder.

**The Root Cause:** 
Both automated AI agents and the user have previously developed the habit of placing temporary "throwaway" scripts directly in the root directory rather than within the designated `scratch/` or `tmp/` folders. Furthermore, the original scanning parameters of the `/junk` skill were limited and did not proactively target prefixes like `test_` or `check_`, leading to a mass accumulation of dormant scripts.

## 🧠 Implementation Strategy / Rationale
To permanently resolve root accumulation while maintaining flexibility for developers, we implemented a **Compartmentalization + TTL (Time to Live)** strategy across three architectural tiers:

1. **Systemic Prevention (The Rules):**
   - Appended a strict Compartmentalization Rule to `self-improving/memory.md` ensuring all future AG agents mandate test file creation inside `scratch/` or `tmp/`.
   - Appended matching instructions to `.cursorrules` so external IDE assistants naturally follow the same placement hierarchy.
2. **Aggressive Root Scanning (Layer A):**
   - Upgraded `docs/skills/junk.md`'s regex matching to include `check_*`, `test_*`, `scratch-*`, `*.p1bak`, and ad-hoc patches.
   - Any matches in the root directory are flagged for immediate deletion (Zero Tolerance).
3. **Forgiving Scratch Scanning (Layer B):**
   - For files located legitimately inside `scratch/` and `tmp/`, we implemented a 3-Day expiration check (`Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-3) }`).
   - This ensures recent, active debug snippets are preserved to maintain valuable recent context, automatically pruning them once they go stale.
