# Unified AI Gateway Hotfix: Google Telemetry Parsing

## 📌 Context
Following the deployment of the Unified AI Gateway (`V7.0.0`), a full end-to-end `/posttest` was executed to verify usage reporting on the production dashboard. The functional test highlighted two data discrepancies when interfacing specifically with `generativelanguage.googleapis.com` API endpoints:
1. **Model Identification Issue**: Dashboard entries logged as `unknown-model`.
2. **Token Registration Edge Case**: Requests processed non-streaming generation configs displayed `0` output tokens despite legitimate metric data existing on the JSON response payload. 

## 🚀 Execution Summary

### 1. Model Parsing URL Extraction
Unlike `openai` compatibility standards (OpenRouter, Kie.ai) which typically serialize the model within the POST request body (`parsedBody.model`), Google’s native SDK endpoints encode the target model string within the endpoint URL URI path itself.
- **Fix Applied**: Updated `ai-gateway.js` proxy route handling to dynamically execute a RegEx string extraction mapping against the raw `req.url`:
  ```javascript
  const modelMatch = req.url.match(/\/models\/([^:/?]+)/);
  if (modelMatch) modelFromUrl = modelMatch[1];
  ```

### 2. Output Token Verification 
Audited the `candidatesTokenCount` capturing sequence to ensure strict downstream variable mapping inside the `!isStream` path payload capture hook, successfully verifying that output values from Google accurately assign onto internal memory state variables (`promptTokens` / `completionTokens`).
- Re-executed full PM2 daemon restarts via `pm2 restart ai-gateway`.

## 🔬 E2E Outcomes
The OpenClaw Dashboard successfully presented updated metrics mapping to native UI charts:
- `AI Monitor`: Correctly tracked against `Gemini 2.5 Flash`. 
- `posttest-runner`: Model accurately labeled as `Gemini 2.5 Flash`, output token capture mapped sequentially. 

The proxy telemetry logic is fully validated and accurate across all ecosystem endpoints.
