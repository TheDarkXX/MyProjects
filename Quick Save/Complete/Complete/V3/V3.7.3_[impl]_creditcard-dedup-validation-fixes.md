# V3.7.3 Credit Card Aggressive Dedup & Validation Fixes

**Date:** 2026-04-10
**Status:** ✅ Complete

## 📌 Context
ปัญหาจากการดึงข้อมูลบัตรเครดิต KBank และ Krungsri รอบเดือนมีนาคม 2026 พบว่ามีปัญหาหลัก 2 อย่างที่ทำการแก้ไขไปแล้วในวันนี้:
1. **KBank:** ข้อมูลเข้าซ้ำซ้อน (Duplicate Records) เนื่องจากชื่อที่ OCR สกัดได้ในแต่ละครั้งมีการเรียงคำต่างกันเล็กน้อย เช่น "CDS-Festival" vs "Central Department Store" ทำให้ Dedup Key เดิม (อิงจาก Name) มองว่าเป็นคนละรายการ
2. **Krungsri:** AI ทำการสลับช่องระหว่างชื่อรายการ (Name) และจำนวนเงิน (Price) หรืออ่านตัวเลขติดกันจนทำให้ช่องราคากลายเป็นค่าว่างหรือผิดปกติ

---

## 🏗️ Architectural Decisions & Implementation Details

เพื่อแก้ปัญหาแบบเบ็ดเสร็จ (100% Foolproof) ได้มีการปรับเปลี่ยน Logic ดังนี้:

### 1. [KBank] Aggressive Deduplication Strategy
**Decision:** เปลี่ยนจาก `date + name + price` เป็น `date + price` สำหรับสร้าง Dedup Key เพื่อป้องกันปัญหาที่ AI อ่านชื่อพลาด หรือตั้งชื่อต่างกัน
- **ไฟล์:** `credit-card-agent.js`
- **โค้ดที่ปรับใหม่:**
  ```javascript
  const normName = (item.name || '').toLowerCase().replace(/[^a-z0-9]/g, '').slice(0, 15);
  // ใช้แค่ Bank + Date + Price บวกกับ normName สั้นๆ ให้ครอบคลุมที่สุด
  source_ref: `cc-dedup-${bankDisplay}-${finalDate}-${normName}-${Math.abs(Number(item.price)||0)}`
  ```

**Data Correction Script:** มีการเขียนสคริปต์กวาดล้าง DB บน VPS ปัจจุบันโดยเจาะจงเฉพาะ `date + price` ที่ซ้ำกัน แล้วทำการลบ Row ที่ Insert มาทีหลังออก
- **ผลลัพธ์:** ลบข้อมูลซ้ำของ KBank ออกไปจำนวน **7 รายการ** เช็คแล้วเหลือแค่ 12 รายการที่ Unique

### 2. [Krungsri] Name-Price Swap Validation Guard
**Decision:** ใส่ Guard เพิ่มใน `validation-utils.js` (หมวด Auto-Fix) หากพบว่า Field `name` กลายเป็นตัวเลขล้วน ให้ทำการสลับค่า Name กับ Price ทันที และกันเหนียวไม่ให้แสกนเจอ Barcode หรือตัวเลขที่สูงเกินไปมาเซฟลง DB
- **ไฟล์:** `validation-utils.js`
- **Snippet หลีกเลี่ยง Name หลุดมาเป็นราคาสลับกัน:**
  ```javascript
  // Guard: Name vs Price swap detection (OCR Bug)
  let testName = (current.name || '').toString().trim().replace(/,/g, '');
  if (/^[\d\.]+$/.test(testName) && parseFloat(testName) > 0) {
     let potentialName = current.description || 'Unknown';
     current.price = parseFloat(testName);
     current.name = potentialName;
     isFixed = true;
  }
  
  // Safety check extremely high price (Barrier)
  if (amt > 500000) {
     logValidationEvent('cc-agent', 'DROP', 1, `Suspiciously high amount ${amt} THB - might be a barcode OCR error`);
     continue; // skip this transaction
  }
  ```

### 3. [Overall] AI Prompt Enforcement
ทำการอัพเดทย้ำกฎในการดึงข้อมูลเข้าไปยัง AI ตรงๆ ใน `credit-card-agent.js` เพื่อบังคับ Strict Format การอ่านคอลัมน์ของ Krungsri (ป้องกันการสลับตั้งแต่ต้นทาง) และการดึงข้อมูล KBank ให้ได้ครบทุกบรรทัดแม้ว่ารูปแบบสเตทเม้นต์จะพับครึ่งมา:
```javascript
## Bank Specific Rules:
- **KBank**: Some transactions might span multiple lines or have installment patterns (e.g., 02/06). Extract the full meaning and DO NOT skip installment items.
- **Krungsri**: The vision OCR table column order is strict: Date | Description (Payee) | Amount (THB) | Credit/Debit. NEVER swap the Name and Amount fields. If you see a number in the Name field, you have made a mistake.
```

## 🚀 Post-Investigation & Root Cause Analysis (2026-04-11)
จากการตรวจสอบปัญหา "Krungsri สลับ/ย้ำซ้อน" และ "SCB ซ้ำ 2 รายการ" พบว่าปัญหาที่แท้จริงเกิดจาก **Date Corruption Bug** ใน `validation-utils.js`:
- Logic เดิมใช้ `current.date.replace(year, currentYear)` ทำให้วันที่พัง (เช่น `2026-03-02` กลายเป็น `2026026-03-02`)!
- เมื่อวันที่เปลี่ยนไปเป็นค่าแปลกๆ มันจะข้ามกลไก Deduplication `source_ref` ไปอย่างสิ้นเชิง ทำให้ไอเทมเดิมถูก Insert ใหม่ซ้ำๆ (ยอดซ้ำ SCB ถือกำเนิดขึ้นจากสิ่งนี้)
- สำหรับ Krungsri การเรียงแถว(Row Alignment) ใน Prompt ถูกเพิ่มกำชับว่าห้ามอ่านสลับแถวกันเด็ดขาด และซ่อม `regex` ตรวจสอบ Price-Swap ให้รวมคำว่า `THB` ไปด้วย
- รันลบข้อมูลวันที่พังๆ ที่ค้างในระบบทั้งหมดเรียบร้อย (`length(date) != 10`) ตอนนี้ Data บน VPS Clean แล้ว
