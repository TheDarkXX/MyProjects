# V3.8.0 OpenClaw Token Analytics Optimization & Heartbeat Fix

## Context
The goal was to refine the OpenClaw Dashboard token usage reporting to ensure 100% accuracy, strict adherence to the project's Design System, and resolve excessive token burn caused by Web UI agent connections.

## Key Changes
1. **Dynamic Session Date Filtration:** Upgraded the JSON session parser to accurately reconstruct daily tokens bound precisely between `00:00:00` and `23:59:59` to fix data loss (especially the "Yesterday" filter bug).
2. **Design System & Cache Visualization:**
   - Separated "Cache Read" tokens completely from "Billed Inputs" at the pipeline level to ensure all analytic cards and donut charts reflect pure paid usage.
   - Applied the strict 3-Stop Gradient rule for Input (Violet P1), Output (Hot Pink P2), and Cache (Burnt Orange P3).
   - Added an interactive click-to-hide `Legend` for both the Daily and Hourly Usage graphs, defaulting Cache Context to "Hidden".
   - Fixed buggy tooltips on Hourly charts so they display the correct dataset label ("Cache Context: 2.9M").
3. **Control UI Heartbeat Inflation Fix:**
   - Diagnosed 2.8M token usage spike resolving to automatic Web UI reconnects triggering full `/root/.openclaw/workspace/HEARTBEAT.md` script checks (~200K tokens each).
   - Completely scrubbed automated checking logic from the Heartbeat config. Web UI pollers now respond strictly with "HEARTBEAT_OK", saving virtually all wasteful tokens.
   - Left scheduled and proactive reporting up to the dedicated `cron jobs` for total isolation.
4. **Token Pricing Models:**
   - Updated the mapping for Discord-reported models (`Gemini 2.5 Flash DC` & `Gemini 2.5 Pro DC`) to correctly latch onto Kie.ai's updated Pay-per-token API prices ($0.09/$0.75 and $0.38/$3.00 respectively), finally successfully activating the "Estimated Cost" metric layout.
