# Quick Save ↔ Dev Lab Sync — Implementation Plan

## Problem Statement

Currently two separate systems track dev work but don't talk to each other:

| System | Location | Purpose | Data |
|--------|----------|---------|------|
| **Quick Save** | Local Git → VPS (`Quick Save/`) | Implementation plans, studies, docs | Rich YAML frontmatter + full markdown body |
| **Dev Lab** | Brain App DB (SQLite `tasks` table) | Dev task tracking (Kanban) | Simple tasks: title, status, priority, description |

**Result:** Quick Save plans are invisible on the web dashboard. Dev Lab tasks have no link to their implementation documentation. You can't see the "big picture" of all plans + tasks in one place.

---

## Goal

Make Quick Save plans visible and manageable from the Dev Lab dashboard — **without breaking either system's existing workflow**.

---

## Architecture Decision: File-First, One-Way Display + Action

> [!IMPORTANT]
> **Quick Save files are the source of truth.** The Dev Lab page will *read* them via API, not duplicate them into the DB. This prevents data drift and conflicts.

```
Quick Save Files (Git)          Dev Lab Page (Browser)
┌─────────────────────┐         ┌──────────────────────────┐
│ Active/              │──scan──→│ 📋 Plans Tab              │
│ Complete/            │         │  ┌─ Active (5)           │
│ Rejected/            │         │  ├─ Complete (21)        │
│ Icebox/              │         │  ├─ Icebox (1)           │
│                      │←─move──│  └─ Rejected (1)         │
│ (YAML frontmatter)  │         │                          │
└─────────────────────┘         │  [Change Status] button  │
                                │  → calls API → moves file│
                                └──────────────────────────┘
```

**Why NOT bi-directional DB sync?**
- Quick Save files have 15+ metadata fields that don't map to the simple `tasks` table
- Version numbers, conversation IDs, related files, full markdown body — none of this fits in a task row
- Copying data into DB creates a stale copy that drifts from the file
- File-first means AG can edit files directly and changes appear on refresh

> [!NOTE]
> **Web UI is primarily a read-only viewer.** The real work (creating plans, updating status, editing content) happens through AG during active development sessions. AG auto-updates files as tasks are completed. The web UI is for **monitoring progress and quick status overview** — not primary editing.

## Proposed Changes

### Phase 1: Backend — Plans API

#### [NEW] `routes/plans.js`

New Express router that scans the Quick Save directory tree on the VPS filesystem.

**Endpoints:**

| Method | Route | Description |
|--------|-------|-------------|
| `GET` | `/api/plans` | Scan all Quick Save folders, parse YAML frontmatter, return JSON array |
| `GET` | `/api/plans/:filename` | Get single plan's full content (frontmatter + body) |
| `PUT` | `/api/plans/:filename/status` | Move file between folders (change status) |

**GET `/api/plans` Response Shape:**
```json
{
  "data": [
    {
      "filename": "V4.1.0_[study]_qwen3-moe-model-migration.md",
      "folder": "Complete",
      "version": "4.1.0",
      "type": "study",
      "status": "complete",
      "outcome": "shipped",
      "date": "2026-04-11",
      "tags": ["ai", "model", "qwen"],
      "summary": "Comprehensive evaluation...",
      "conversation": "16b22b8f-...",
      "conversation_title": "Optimizing AI Model Migration",
      "brain_task_id": null,
      "related": ["docs/agent_model_config.md", "..."]
    }
  ],
  "stats": {
    "active": 5,
    "complete": 21,
    "rejected": 1,
    "icebox": 1,
    "total": 28
  }
}
```

**YAML Parsing:** Use simple regex-based frontmatter extraction (no new dependency needed). The frontmatter is always between `---` delimiters.

**Legacy Folders:** Also scan `Complete/V.1/` and `Complete/V2/` subdirectories recursively. These contain older plans without YAML frontmatter — extract what we can from filenames and show them as "legacy" entries with minimal metadata.

**PUT `/api/plans/:filename/status` Request:**
```json
{
  "newStatus": "complete"   // active | complete | rejected | icebox
}
```

Action: `fs.renameSync()` the file from its current folder to the target folder. Update the `status:` field in the YAML frontmatter. Git commit + push automatically.

## 🔥 Post-Launch Update: Unified Pipeline Refactor [✅ DONE]

After initial implementation, user requested to **merge** the Tasks and Plans into a single, unified pipeline rather than keeping them in separate UI views.

**Why:** The standard Tasks should behave as the initial "Idea Backlog" (originating from Discord Bot), and those ideas eventually hatch into Markdown implementation plans dynamically. 

**Changes Made:**
1. **Removed the "Plans" tab toggle.** Kept only `List View` and `Pipeline (Board)`.
2. **Unified Board (5 Columns):**
   - 💡 **Idea Backlog**: standard `tasks` hitting DB (`todo`).
   - 🟡 **Active Plans**: `plans` returning from filesystem (`active`).
   - 🧊 **Icebox**: `plans` on hold (`icebox`).
   - ✅ **Complete**: mixed collection of `done` tasks and `complete` plans.
   - ❌ **Rejected**: mixed collection of `cancelled` tasks and `rejected` plans.
3. **Smart Drag-and-Drop (`updateItemStatusUnified`):**
   - Implemented dynamic status updater that checks if the dropped ID is an integer (DB task) or `.md` string (QS plan), and fires the appropriate API logic. 
4. **Unified List View:**
   - Unified `devlabTasks` and `dlPlans` into one dataset sorted by date.
   - Rendered using distinct `💡`/`📄` icons to separate the context while keeping them visible side by side.

**Modal Layout:**

```
┌──────────────────────────────────────────────────────────────┐
│  📋 V4.1.0 [study] Qwen3 MoE Model Migration          [✕]  │
│  ─────────────────────────────────────────────────────────── │
│  Status: ✅ Complete  │  Outcome: 🚀 Shipped  │  Apr 11     │
│  Tags: ai, model, qwen, migration, openrouter               │
│  Conversation: "Optimizing AI Model Migration"               │
│  Related: docs/agent_model_config.md, openclaw.js, ...       │
```

Uses the `GET /api/plans/:filename` endpoint to fetch full content. Renders markdown body into HTML via `marked.parse(body)` on the client side. Tables, code blocks, headings, and lists all render beautifully with dark-theme CSS.

## Conflict Prevention Strategy

| Scenario | Resolution |
|----------|------------|
| AG edits file while user views on web | Next refresh shows latest (file is source of truth) |
| User changes status on web | API moves file + updates frontmatter + git commits |
| Two users change status simultaneously | `fs.renameSync` is atomic; second call sees file is already moved |
| File has no YAML frontmatter (legacy V1/V2) | Shown as "legacy" card with filename-only metadata, read-only |
| AG updates file while user views on web | File is truth — next refresh shows latest. No lock needed |
| Status changed on web | API moves file, updates frontmatter, auto-commits + pushes to git |
