# 🚀 Implementation Plan: AG Skills Engine

This document captures the full context of the Hermes Agent evaluation, the realization that our system is already highly mature, and the resulting plan to build our own Skills Engine instead of migrating.

---

## 1. Context & Architectural Decisions (Hermes Agent Deep Study)

### 🔍 Initial Investigation
Boss asked to evaluate **Hermes Agent** (and the Hermes Atlas ecosystem) to see if it's better than OpenClaw, how it applies to our system, and if we should migrate to it to solve system instability.

### 💡 Key Findings from Codebase Audit
1. **Hermes Architecture:** Hermes is OpenClaw's official successor. It features a robust Python-based loop, a progressive disclosure skills system that auto-creates `SKILL.md` files upon task completion, and a sophisticated cross-platform gateway (15+ platforms).
2. **OpenClaw Gateway is Dead Weight:** We audited `discord-bot/lib/ai.js` and confirmed that our system is already highly decoupled from OpenClaw. Our AI agents (Pinky, Tata, Lily) use `fetch()` to make **direct API calls** to Google AI Studio, OpenRouter, and Kie.ai. The actual OpenClaw Gateway Service (port 18790) is virtually unused in practice.
3. **Kie-Proxy (Port 18800) Independence:** The `kie-proxy.js` service is a custom solution to strip unsupported fields for Kie.ai and is completely independent of the OpenClaw runner.
4. **Multi-Agent Orchestration:** We already have advanced multi-agent workflows running natively in Node.js (e.g., `scripts/agent-combat.js` with Doctor Skill and Nong Juji), circumventing the need for an external orchestration framework.
5. **No Need for Hermes Gateway:** Since we exclusively use Discord and have a tailored Hono/Node.js stack optimized for our specific pipelines (Credit Card, Content, Health News), injecting a heavy Python-based framework like Hermes would introduce new tech debt and become another "dead weight."

### 🎯 The "Aha!" Moment & Strategic Decision
Hermes has one killer feature we lack: **The Auto Skills System**. It patches itself and learns autonomously. Our AG has the *infrastructure* for this (the new Knowledge Items sysytem), but lacks the *automatic loop*.

**Final Decision:**
1. **REJECT Migration:** We will NOT migrate to Hermes.
2. **PURGE OpenClaw:** We will completely uninstall `openclaw-runner.mjs` and remove port 18790 to clean up the architecture. (*Deferred to V4.6.0 Icebox*)
3. **STEAL The Best Feature:** We will build a native **3-Layer AG Skills Engine** to replicate Hermes' auto-learning feature entirely inside our existing Node.js ecosystem.

## 3. Phase 2: AG Skills Engine Architecture

We need AG to emit skill data *during* the conversation, then compile it into a KI artifact *after* the push.

### The 3-Layer Skills Engine Flow

```
┌──────────────────────────────────────────────────────────────┐
│  LAYER 1: AG Runtime (Prompt-driven, ~85% reliable)          │
│                                                              │
│  ┌─────────┐    ┌──────────────┐    ┌───────────────────┐   │
│  │ Start   │───▶│ Load Skills  │───▶│ Work on Task      │   │
│  │ Convo   │    │ (KI auto)    │    │ (use skills)      │   │
│  └─────────┘    └──────────────┘    └────────┬──────────┘   │
│                                              │               │
│                                   ┌──────────▼──────────┐   │
│                                   │ Task Complete?       │   │
│                                   │ Complex (5+ edits)? │   │
│                                   └──────────┬──────────┘   │
│                                              │ Yes           │
│                                   ┌──────────▼──────────┐   │
│                                   │ EMIT skill-emit.jsonl│   │
│                                   │ (append to ledger)   │   │
│                                   └─────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│  LAYER 2: Post-Push Compiler (Code-driven, catches misses)   │
│                                                              │
│  Trigger: runs inside deploy.js push AFTER git push vps      │
│                                                              │
│  1. Read self-improving/skill-emit.jsonl                     │
│  2. For each unprocessed entry:                              │
│     a. Call AI → refine into structured skill                │
│     b. Check if existing KI covers this → update or create   │
│     c. Write to knowledge/ directory                         │
│  3. Mark entries as processed                                │
│  4. Check recent conversation metadata for missed emits      │
│  5. Report new/updated skills to console                     │
└──────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│  LAYER 3: Self-Repair Loop (Code-driven, detect failures)    │
│                                                              │
│  Trigger: AG emits "skill-fail" entry to ledger              │
│           + Post-Push Compiler detects and processes          │
│                                                              │
│  1. AG uses a skill → task fails → AG emits failure entry    │
│  2. Failure entry includes: skill_id, what_failed, fix       │
│  3. Compiler reads failure → patches KI artifact             │
│  4. Next conversation → AG loads updated skill               │
└──────────────────────────────────────────────────────────────┘
```

## 4. Design Details

### Layer 1: AG Rules (AGENTS.md)

Add new Iron Rule section:

```markdown
## ⛔ Skills Engine — Iron Rule (NEVER violate!)

### After Complex Task Completion (5+ file edits or tool calls)
AG MUST append a skill emit entry to `self-improving/skill-emit.jsonl`:

Format (one JSON object per line):
{"ts":"ISO","type":"skill","conv":"conversation-id","trigger":"when to use","steps":["step1","step2"],"anti_patterns":["what not to do"],"tags":["tag1"],"files_touched":["file1.js"],"summary":"what was accomplished"}

### After Using an Existing Skill (KI) That Led to Failure
AG MUST append a failure entry:

{"ts":"ISO","type":"skill-fail","conv":"conversation-id","skill_id":"ki-folder-name","what_failed":"description","root_cause":"why","fix":"what should change"}

### Rules:
1. EMIT before reporting completion to user
2. Keep entries concise — max 500 chars per field
3. Tags should match KI folder naming convention (lowercase-with-dashes)
4. Files touched = actual files edited, not all files viewed
```

### Layer 2: Skills Compiler Script
New file: `scripts/compile-skills.js`

```javascript
// Pseudocode — actual implementation

import { readFileSync, writeFileSync, existsSync, mkdirSync } from 'fs';
import { join, resolve } from 'path';
import { homedir } from 'os';

const LEDGER = resolve('self-improving/skill-emit.jsonl');
const KI_DIR = join(homedir(), '.gemini', 'antigravity', 'knowledge');

async function compileSkills() {
  if (!existsSync(LEDGER)) return;
  
  const lines = readFileSync(LEDGER, 'utf-8').split('\n').filter(Boolean);
  const entries = lines.map(l => { try { return JSON.parse(l); } catch { return null; } }).filter(Boolean);
  const unprocessed = entries.filter(e => !e.processed);
  
  if (unprocessed.length === 0) return;

  // Group by tags to find related skills
  const grouped = {};
  for (const entry of unprocessed) {
    const key = entry.tags?.[0] || 'general';
    if (!grouped[key]) grouped[key] = [];
    grouped[key].push(entry);
  }

  for (const [tag, entries] of Object.entries(grouped)) {
    const kiDir = join(KI_DIR, `skill-${tag}`);
    const artifactDir = join(kiDir, 'artifacts');
    
    if (entries[0].type === 'skill-fail') {
      await repairSkill(kiDir, entries);
    } else {
      await upsertSkill(kiDir, artifactDir, tag, entries);
    }
  }

  // Mark all as processed
  const processed = lines.map(l => {
    try {
      const obj = JSON.parse(l);
      obj.processed = true;
      return JSON.stringify(obj);
    } catch { return l; }
  });
  writeFileSync(LEDGER, processed.join('\n') + '\n');
}

async function upsertSkill(kiDir, artifactDir, tag, entries) {
  mkdirSync(artifactDir, { recursive: true });
  
  const prompt = `You are a skill compiler. Given these raw task completions, create a structured reusable skill document.
Raw entries: ${JSON.stringify(entries, null, 2)}`;

  const skillContent = await callAI(prompt);
  writeFileSync(join(artifactDir, 'skill.md'), skillContent);

  const metadata = {
    title: `Skill: ${tag}`,
    summary: entries.map(e => e.summary).join('. '),
    references: entries.flatMap(e => (e.files_touched || []).map(f => ({ type: 'file', value: f })))
  };
  writeFileSync(join(kiDir, 'metadata.json'), JSON.stringify(metadata, null, 2));
}

async function repairSkill(kiDir, failEntries) {
  const skillPath = join(kiDir, 'artifacts', 'skill.md');
  if (!existsSync(skillPath)) return;
  
  const currentSkill = readFileSync(skillPath, 'utf-8');
  const prompt = `This skill was used but FAILED. Update it with the fixes.
Current Skill: ${currentSkill}  
Failure Reports: ${JSON.stringify(failEntries, null, 2)}`;

  const updatedSkill = await callAI(prompt);
  writeFileSync(skillPath, updatedSkill);
}

async function callAI(prompt) {
  const apiKey = process.env.GOOGLE_API_KEY;
  const res = await fetch('https://generativelanguage.googleapis.com/v1beta/openai/chat/completions', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${apiKey}` },
    body: JSON.stringify({ model: 'gemini-2.5-flash', messages: [{ role: 'user', content: prompt }] })
  });
  const data = await res.json();
  return data.choices?.[0]?.message?.content || '';
}

compileSkills().catch(console.error);
```

## 5. Why This Is Better Than Hermes

| Feature | Hermes Skills | AG Skills Engine |
|---------|---------------|------------------|
| **Storage** | Local SKILL.md files | Knowledge Items (structured) |
| **Trigger** | Auto (code-level) | Hybrid: prompt + code fallback |
| **Repair** | Same session only | Cross-session (repair KI for next chat) |
| **Multi-PC sync** | No | Yes (KI directory syncs via .gemini) |
| **Human oversight** | None | skill-emit.jsonl is readable, editable |
