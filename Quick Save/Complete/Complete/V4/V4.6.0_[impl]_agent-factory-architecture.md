## สรุปสั้น (TL;DR)

**ยังไม่มี framework ที่ทำ "scan cron → สร้าง agent อัตโนมัติ" แบบ plug-and-play**
แต่ pattern ที่ industry ใช้กันแล้วคือ **"Spec-to-Agent"** — เขียน config YAML/JSON → runtime อ่านแล้ว spawn agent ให้เอง

สำหรับ stack ของเรา (Node.js / VPS / SQLite / kie-proxy) ทางที่ดีที่สุดคือ **สร้าง lightweight orchestrator ของเราเอง** ที่ build on top of สิ่งที่มีอยู่แล้ว (Watchdog V2 + CronTracker + agent SOUL files)

## 1. Industry Landscape — มีอะไรอยู่แล้ว?

### Frameworks ที่ทำใกล้เคียงที่สุด

| Framework | ภาษา | จุดเด่น | จุดอ่อนสำหรับเรา |
|-----------|-------|---------|-------------------|
| **CrewAI** | Python | YAML-based agent definition, role-based crews | Python-only, overkill สำหรับ cron scripts |
| **LangGraph** | Python | Stateful graph workflows, checkpoint/resume | Python, learning curve สูง |
| **AutoGen** | Python | Multi-agent conversation, async | Python, ซับซ้อนเกินไป |
| **Trigger.dev** | TypeScript | Durable execution, built-in cron, retries | ต้อง self-host platform ทั้งตัว |
| **n8n** | TypeScript | Visual workflow, drag-and-drop | Visual-first, ไม่เหมาะกับ code-heavy agents |
| **Temporal** | Any | Enterprise-grade durable execution | Heavy infra, overkill สำหรับ VPS เดียว |

### Emerging Standards

| Spec | สถานะ | หมายเหตุ |
|------|--------|----------|
| **Open Agent Specification** | Early stage | Framework-agnostic YAML for agents & flows |
| **Agent Format (.agf.yaml)** | Proposal | "Kubernetes for agents" — full agent lifecycle |
| **Microsoft Agent Framework** | Active | Declarative YAML agent definitions |

> **สรุป:** ทุก framework เป็น Python-heavy หรือต้องติดตั้ง platform ใหญ่ ไม่มีอันไหนที่ "scan crontab แล้ว auto-spawn agent" โดยตรง

## 2. The "Spec-to-Agent" Pattern — วิธีที่ Industry ใช้จริง

```
┌─────────────┐     ┌──────────────┐     ┌───────────────┐
│  Job Specs   │────▶│  Orchestrator │────▶│  Agent Runtime│
│  (YAML/JSON) │     │  (Scanner)    │     │  (Executor)   │
└─────────────┘     └──────────────┘     └───────────────┘
       │                    │                      │
  - schedule               - reads specs          - single agent
  - goal                   - detects changes       - or multi-agent crew  
  - tools needed           - decides single/orch   - runs with tools
  - agent type             - spawns & monitors     - reports results
```

### Core Idea
แทนที่จะ hardcode agent ทุกตัว ให้เขียน **spec file** แล้วมี **orchestrator** อ่าน + สร้าง agent ให้อัตโนมัติ

## 3. Recommended Architecture สำหรับ Stack เรา

### ทำไมถึงแนะนำ "Build Native" แทน Adopt Framework

| เหตุผล | รายละเอียด |
|--------|------------|
| **Stack match** | เราเป็น Node.js ทั้งหมด — framework ส่วนใหญ่เป็น Python |
| **มีพื้นฐานดีแล้ว** | Watchdog V2, CronTracker, agent SOUL files, kie-proxy — แค่ต่อจิ๊กซอว์ |
| **VPS เดียว** | ไม่ต้องการ Kubernetes/Docker orchestration |
| **ความง่าย** | 1 file scanner + 1 spec schema + 1 executor = จบ |

### Proposed Architecture: "CronAgent Registry"

```
cron-registry/
├── specs/                          ← Agent Spec Files (YAML)
│   ├── gmail-expenses.yaml         
│   ├── content-auto-planner.yaml   
│   ├── daily-review.yaml           
│   ├── watchdog-health.yaml
│   └── cc-statement-scan.yaml      
├── orchestrator.js                 ← Main: scan → detect → spawn
├── executor.js                     ← Runtime: single or orchestrated
└── registry.db (or .json)          ← State tracking
```

### Agent Spec Format (YAML)

```yaml

id: gmail-expenses
name: "Gmail Expense Scanner"
description: "Scan Gmail for receipts and credit card statements"

  model: qwen3-235b
  system_prompt_ref: "config/SOUL-brain-worker.md"
  tools:
    - gmail_api
    - sqlite_write
    - pdf_extract

  retry:
    max_attempts: 3

  track_tokens: true
```

### Orchestrator Logic (Pseudocode)

```javascript
// orchestrator.js — Core loop
async function scanAndExecute() {
  // 1. Scan all spec files
  const specs = loadAllSpecs('./cron-registry/specs/*.yaml');
  
  // 2. For each spec, check if it should run now
  for (const spec of specs) {
    if (!shouldRunNow(spec.schedule)) continue;
    if (isAlreadyRunning(spec.id)) continue;
    
    // 3. Decide: single agent or orchestrated?
    if (spec.type === 'single') {
      await executeSingleAgent(spec);
    } else if (spec.type === 'orchestrated') {
      await executeOrchestrated(spec);
    }
    
    // 4. Log execution to CronTracker
    await cronTracker.log(spec.id, result);
  }
}

async function executeSingleAgent(spec) {
  const response = await callAI({
    model: spec.agent.model,
    systemPrompt: loadFile(spec.agent.system_prompt_ref),
    tools: resolveTools(spec.agent.tools),
    headers: { 'X-Claw-Cron': spec.monitoring.cron_header }
  });
  return response;
}

async function executeOrchestrated(spec) {
  const context = {};
  for (const agentSpec of spec.orchestration.agents) {
    // Inject previous agent's output
    const input = agentSpec.input_from 
      ? getFromContext(context, agentSpec.input_from) 
      : null;
    
    const result = await callAI({
      model: agentSpec.model,
      systemPrompt: loadFile(agentSpec.system_prompt_ref),
      tools: resolveTools(agentSpec.tools),
      userMessage: input,
    });
    
    context[`${agentSpec.id}.${agentSpec.output_key}`] = result;
  }
  return context;
}
```

## 4. Decision Matrix: เลือกทางไหนดี?

### Option A: Build Native (แนะนำ ⭐)

| Pro | Con |
|-----|-----|
| ต่อยอดจากที่มีอยู่ (Watchdog, CronTracker) | ต้องเขียนเอง ~500 lines |
| Node.js native, ไม่ต้อง install อะไรใหม่ | ไม่มี visual editor |
| เข้าใจง่าย, debug ง่าย | ไม่มี community support |
| เหมาะกับขนาด project (8-10 cron jobs) | |

**Effort:** ~1-2 วัน  
**Complexity:** ★★☆☆☆

### Option B: Adopt n8n (Self-hosted)

| Pro | Con |
|-----|-----|
| Visual workflow editor | ต้อง install Docker + n8n server |
| Drag-and-drop agent flows | เพิ่ม resource ใน VPS |
| Built-in cron + retry | Learning curve ใหม่ |
| Community plugins | Overkill สำหรับ 10 jobs |

**Effort:** ~2-3 วัน setup, ongoing maintenance  
**Complexity:** ★★★☆☆

### Option C: Adopt CrewAI (Python)

| Pro | Con |
|-----|-----|
| Mature multi-agent framework | Python → แยก stack |
| YAML-based agent definition | ต้อง install Python runtime |
| Active community | Overhead สูง |

**Effort:** ~3-5 วัน + Python migration  
**Complexity:** ★★★★☆

## 6. Key Best Practices

1. **Start Small** — อย่าเริ่มจาก orchestrate ทุกอย่าง เริ่มจาก single agent + existing cron
2. **Spec = Source of Truth** — ทุก job ต้องมี spec file, ไม่มี "ลอย" ใน crontab
3. **Validate AI Output** — ทุก agent ต้องผ่าน validation ก่อน save (กฎเหล็กที่มีอยู่แล้ว)
4. **Observability First** — ทุก execution ต้อง log ลง CronTracker + token tracking
5. **Graceful Degradation** — ถ้า orchestrator ล่ม, cron jobs เดิมต้องยังทำงานได้
6. **Don't Over-Orchestrate** — ถ้า job เป็น simple script (backup, cleanup) ไม่ต้องยัดเป็น agent

## 7. สรุปคำตอบ

> **มีคนทำแนวนี้ไหม?** — มี แต่เป็น Python-heavy (CrewAI, LangGraph) หรือ platform ใหญ่ (n8n, Temporal) ไม่มี "magic button" ที่ scan cron แล้ว auto-create agent
>
> **แนวทางแนะนำ?** — **Build native "CronAgent Registry"** ต่อยอดจาก Watchdog V2 + CronTracker ที่มีอยู่ ใช้ YAML spec files เป็น source of truth + Node.js orchestrator ธรรมดาๆ
>
> **ง่าย แม่นยำ best practice?** — ใช้ "Spec-to-Agent" pattern (YAML → orchestrator → executor) ซึ่งเป็น pattern เดียวกับที่ CrewAI, Microsoft Agent Framework ใช้ แต่เขียนเองใน Node.js ให้เข้ากับ stack

```javascript
  const context = {
    available_tools: Object.entries(TOOL_REGISTRY).map(([id, t]) => 
      `${id}: ${t.description} [risk: ${t.risk}]`
    ).join('\n'),
    
    known_patterns: Object.entries(PATTERNS).map(([id, p]) =>
      `${id}: ${p.description} (ปกติใช้ ${p.typical_agents} agent${p.typical_agents > 1 ? 's' : ''})`
    ).join('\n'),
    
    existing_crons: await getExistingCrons(),  // ดึง cron ปัจจุบันมาเทียบ
    
    model_catalog: {
      cheap:  { id: "qwen3-30b",  cost: "$0.001/1k tokens", best_for: "simple classification, formatting" },
      smart:  { id: "qwen3-235b", cost: "$0.005/1k tokens", best_for: "complex reasoning, extraction" },
      vision: { id: "qwen3-vl-32b", cost: "$0.003/1k tokens", best_for: "image/PDF reading" }
    }
  };

  // 2. เรียก AI ให้วิเคราะห์
  const blueprint = await callArchitect(userRequest, context);
  
  // 3. Return blueprint ให้ user อนุมัติ
  return blueprint;
}
```

#### System Prompt ของ Meta-Agent (Architect)

```markdown
คุณคือ Agent Architect — ผู้เชี่ยวชาญในการออกแบบ AI Agent Systems

เมื่อ user อธิบายสิ่งที่ต้องการ คุณต้อง:

## Step 1: วิเคราะห์ (Analysis) 
- ระบุว่างานนี้มีกี่ขั้นตอน
- แต่ละขั้นตอนมี risk level เท่าไหร่
- ต้องใช้ tools อะไรจาก Tool Registry
- มี pattern ไหนใน Pattern Library ที่ตรง

## Step 2: ตัดสินใจ Single vs Orchestration
ใช้ Decision Matrix นี้:

| เงื่อนไข | ผลลัพธ์ |
|-----------|---------|
| ขั้นตอนเดียว, risk ต่ำ | → Single Agent |
| หลายขั้นตอน แต่ต่อเนื่องง่ายๆ | → Single Agent + validation step |
| มีขั้นตอน risk สูง (เช่น เขียน DB) | → Orchestration: แยก extractor + validator |
| ต้องการ creative + review | → Orchestration: creator + reviewer |
| ดึงข้อมูลจากหลายแหล่ง | → Orchestration Parallel: agents ทำพร้อมกัน |

## Step 3: ออกแบบ Blueprint
สำหรับแต่ละ Agent ต้องระบุ:
- ชื่อ + หน้าที่ (1 บรรทัด)
- System Prompt (สั้นชัด)
- Tools ที่อนุญาต (จาก Tool Registry เท่านั้น)    
- Model ที่เหมาะสม (cheap/smart/vision)
- Input: รับจากไหน
- Output: ส่งอะไรออกไป
- Validation: ตรวจสอบผลลัพธ์อย่างไร

## Step 4: เสนอ Cron + Monitoring
- แนะนำ cron expression + เหตุผล
- Token cost estimate ต่อรอบ
- Error handling strategy
- Monitoring: ผูก Watchdog อย่างไร

## Step 5: ประเมิน Risk & Cost
- Token cost estimate ต่อวัน/เดือน
- จุดที่อาจพัง + แผนรับมือ
- เทียบกับ patterns ที่มีอยู่ — ซ้ำซ้อนกับ cron ที่มีอยู่แล้วหรือไม่

Return เป็น JSON format ตาม AgentBlueprint schema
```

### Component 4: Builder (ช่างสร้าง — Auto-Generate ทุกอย่าง)

```javascript
// builder.js
export async function buildFromBlueprint(blueprint) {
  const results = { files: [], crons: [], watchdog: [] };
  
  // === 1. Generate Script Files ===
  for (const agent of blueprint.agents) {
    const scriptContent = generateAgentScript(agent, blueprint);
    const filePath = `scripts/agents/${blueprint.id}/${agent.id}.js`;
    fs.writeFileSync(filePath, scriptContent);
    results.files.push(filePath);
  }
  
  // === 2. Generate Orchestrator (ถ้า multi-agent) ===
  if (blueprint.type === 'orchestrated') {
    const orchContent = generateOrchestratorScript(blueprint);
    const orchPath = `scripts/agents/${blueprint.id}/run.js`;
    fs.writeFileSync(orchPath, orchContent);
    results.files.push(orchPath);
  }
  
  // === 3. Register Cron ===
  const cronEntry = `${blueprint.cron.expression} cd /root/brain-app && node ${mainScript} >> /root/brain-app/logs/${blueprint.id}.log 2>&1`;
  await addToCrontab(cronEntry);
  results.crons.push(cronEntry);
  
  // === 4. Register Watchdog Goal ===
  const watchdogGoal = {
    name: blueprint.name,
    priority: blueprint.monitoring.priority,
    checks: [{
      id: `agent-${blueprint.id}-health`,
      description: `ตรวจสอบ ${blueprint.name} ทำงานสำเร็จ`,
      type: 'cron-outcome',
      severity: blueprint.monitoring.severity || 'WARNING',
      autofix: blueprint.monitoring.on_failure || 'alert-only'
    }]
  };
  await registerWatchdogGoal(watchdogGoal);
  results.watchdog.push(watchdogGoal);
  
  // === 5. Dry Run (ถ้าเปิดใช้) ===
  if (blueprint.dry_run) {
    const dryResult = await executeDryRun(blueprint);
    results.dry_run = dryResult;
  }
  
  return results;
}
```

```javascript
// feedback-loop.js
export async function analyzeAgentPerformance(agentId) {
  const logs = await getCronLogs(agentId, { days: 7 });
  
  const metrics = {
    total_runs: logs.length,
    success_rate: logs.filter(l => l.status === 'success').length / logs.length,
    avg_tokens: avg(logs.map(l => l.tokens_used)),
    avg_duration: avg(logs.map(l => l.duration_ms)),
    failures: logs.filter(l => l.status !== 'success')
  };
  
  // AI วิเคราะห์ performance
  const suggestions = await callArchitect(`
    Agent "${agentId}" ทำงานมา 7 วัน ผลลัพธ์:
    - Success rate: ${(metrics.success_rate * 100).toFixed(1)}%
    - Avg tokens per run: ${metrics.avg_tokens}
    - Common failures: ${JSON.stringify(metrics.failures.slice(0,3))}
    
    แนะนำ optimization:
    1. ควรเปลี่ยน model ให้ถูก/แพงกว่านี้ไหม?
    2. ควรแยก/รวม agent ไหม?
    3. ควรปรับ cron schedule ไหม?
    4. มี pattern failure ซ้ำๆ ที่แก้ได้ไหม?
  `);
  
  return { metrics, suggestions };
}
```
