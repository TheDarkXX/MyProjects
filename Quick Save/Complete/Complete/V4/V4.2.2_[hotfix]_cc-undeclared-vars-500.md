# 🐛 500 Internal Server Error on Command Center API

## 🚨 The Issue
The user noticed that the **"cron expense tracker"** hadn't reported since April 8th, and **"Watchdog"** never sent an alert.

Upon inspecting the VPS PM2 logs, we found:
`❌ Error: start_month is not defined`
`--> GET /api/cc/expenses?year_month=2026-04&limit=10 500 3ms`

Due to a copy-paste error in past migrations, these two lines were scattered globally across almost all GET routes (`/events`, `/subscriptions`, `/bills`, `/crons`, `/expenses`, and `/credit-cards/refunds`):
```javascript
if (start_month && start_month.length === 7) start_month += '-01';
if (end_month && end_month.length === 7) end_month += '-31';
```
Since these variables were never declared in `c.req.query()` except in `/credit-cards`, Node.js threw a fatal `ReferenceError`, resulting in a `500 Internal Server Error`.

### Why didn't Watchdog alert?
Watchdog monitors the PM2 process status (`online`, `stopped`, `errored`). Since the `brain-app` process itself never crashed or restarted—only specific endpoints were throwing 500s on the flight—Watchdog saw the app as perfectly healthy (`online`) and thus didn't trigger any alerts.

## 🛠️ The Fix
Removed the undeclared variable logic entirely from all affected routes in `routes/command-center.js`. They were serving no purpose anyway since they weren't being consumed by the SQL queries on those endpoints. They remain properly declared and used solely in the `/credit-cards` endpoint.

## ✅ Verification
The API now reliably handles requests again, and the cron jobs will resume normally starting tonight.
